# mini-kv v896-v920 candidate request package integrity hardening

## Goal

v896-v920 turns the completed v895 candidate document request package into a frozen source fixture and adds a separate integrity evidence chain:

```text
SHARDROUTECANDIDATEREQUESTPACKAGEINTEGRITYJSON
```

The important boundary is that this is not another request echo. Node plan `docs/plans3/v1386-controlled-read-only-shard-preview-candidate-document-request-package-closeout-roadmap.md` says the request-package chain must stop until a reviewed real candidate document exists. This mini-kv batch respects that by auditing the frozen v895 package, its fixture lineage, command exposure, and no-side-effect boundaries. It does not create, import, read, evaluate, accept, reject, approve, sign, execute, route, write, compact, restore, or load any real or synthetic candidate document.

## Module Split

The new code deliberately uses short filenames so the previous long route-preview chain does not keep expanding:

- `include/minikv/shard_preview_candidate_request_package_integrity.hpp` and `src/shard_preview_candidate_request_package_integrity.cpp` own the public command JSON, digest marker, published stage count, and current status.
- `*_integrity_stages` owns the v896-v920 stage catalog. Each stage records `releaseVersion`, `sourceFrozenReleaseVersion`, and `sourceFrozenFixturePath`, so the stage-chain helper can verify continuity.
- `*_integrity_checks` owns the 25 check catalog. The checks cover source fixture freeze, release continuity, count parity, missing/synthetic document boundaries, payload/evaluation/approval/signature/runtime boundaries, write/WAL/router/sibling/archive-walk boundaries, command catalog parity, CLI/TCP smoke parity, and the explicit no-additional-request-echo stop condition.
- `*_integrity_validation` collects the machine-readable pass/fail fields: source package stage count, source chain completeness, integrity check count, published stage count, and all disabled side-effect flags.

This split keeps the core JSON builder from becoming a large catalog file. The catalog data can evolve separately from the formatter and validation rules.

## Command Contract

`src/command.cpp` registers `SHARDROUTECANDIDATEREQUESTPACKAGEINTEGRITYJSON` as a runtime evidence command with no extra arguments. `src/command_contracts.cpp` classifies it as:

```text
category=read
mutates_store=false
touches_wal=false
stable=true
```

`src/line_editor.cpp`, `HELP`, `COMMANDS`, and `COMMANDSJSON` all expose the command. This matters because the command is not just an internal helper hidden inside `SHARDJSON`; it is independently smoke-testable through CLI and TCP while still being read-only.

## SHARDJSON Integration

`src/shard_readiness.cpp` now moves current `releaseVersion` to `v920`, and top-level `status` comes from:

```cpp
shard_preview_candidate_request_package_integrity::candidate_request_package_integrity_status()
```

The old `shardPreviewCandidateDocumentRequestPackageCloseout` section remains in `SHARDJSON` as the v895 source package. The new `shardPreviewCandidateRequestPackageIntegrity` section then points at:

```text
sourceCandidateDocumentRequestPackageReleaseVersion = v895
sourceCandidateDocumentRequestPackageFixturePath = fixtures/release/shard-readiness-v895.json
sourceCandidateDocumentRequestPackagePublishedStageCount = 15
sourceCandidateDocumentRequestPackageRequestItemCount = 15
sourceCandidateDocumentRequestPackageAcceptanceCheckCount = 15
sourceCandidateDocumentRequestPackageGateCount = 38
sourceCandidateDocumentRequestPackageRequestedFieldCount = 20
```

Control-plane consumers should read this as: mini-kv has frozen and audited the request-package evidence chain, but it has not opened document intake. `additionalRequestEchoCreated=false`, `documentIntakeOpened=false`, `realCandidateDocumentRead=false`, `documentRouterInstalled=false`, `writeRoutingAllowed=false`, `touchesWal=false`, and `executionAllowed=false` are the core safety fields.

## Fixture Evidence

The batch first freezes v895:

```text
fixtures/release/shard-readiness-v895.json
```

Then it generates:

```text
fixtures/release/shard-readiness-v896.json
...
fixtures/release/shard-readiness-v919.json
fixtures/release/shard-readiness.json  # current v920
```

The historical fixtures were generated by temporarily setting the stage count and release version for each release, rebuilding `minikv_cli`, running `SHARDJSON` through stdin, extracting only the JSON line, and restoring the final v920 source constants. That makes each fixture's `releaseVersion`, `candidateRequestPackageIntegrityReleaseVersion`, and `sourceFrozenFixturePath` match the stage it represents.

## Tests

New focused tests cover:

- `candidate_request_package_integrity_stages_tests`: stage count, first/last release, source fixture continuity shape.
- `candidate_request_package_integrity_checks_tests`: 25 checks, partial/full JSON clamp behavior, and read-only boundary fields.
- `candidate_request_package_integrity_validation_tests`: validation pass/fail behavior for source package freeze and count alignment.
- `candidate_request_package_integrity_tests`: final command JSON, source v895 linkage, v920 closeout, no intake, no router, no write, no WAL, no execution.
- `candidate_request_package_integrity_fixture_tests`: v895 fixture remains a source package without the new integrity section, while current v920 carries the new integrity section.

Existing `command_tests`, `line_editor_tests`, and `shard_readiness_tests` were updated so the new command is visible in user-facing command surfaces and the top-level `SHARDJSON` status now reflects v920.

## Summary

v896-v920 gives Node, Java, and future mini-kv work a durable versioned integrity handle for the v895 request package, while keeping mini-kv firmly on the evidence side of the boundary: no document intake, no active router, no write path, no WAL touch, and no execution.
