# v33 运行调试归档说明

本轮版本目标：

```text
在 v32 命令总计数基础上，增加按命令类型拆分的统计和命令执行延迟计数。
```

v33 重点验证：

```text
CommandBreakdownMetrics 记录单个命令桶的 total / success / error / latency。
CommandProcessorMetrics 记录全局 total_latency_ns、avg_latency_ns、max_latency_ns。
CommandMetricsTracker 使用 mutex 保证 totals 和 breakdown 快照一致。
未知命令统一归入 UNKNOWN 桶，避免任意错误输入制造无限指标维度。
STATS、HEALTH、server_metrics、tcp_stop 共用 format_command_metrics 输出格式。
代码讲解记录在开发调试同一轮内同步补齐。
```

- 结果：退出码为 0。

## 05-tcp-server-tests.png

- 命令：`.\cmake-build-debug\minikv_tcp_server_tests.exe`
- 结果：退出码为 0。
- 意义：专项验证 TCP server 场景下的共享命令指标：
  - `STATS` 在真实 TCP 请求中输出 `command_breakdown=PING:1/1/0/...`。
  - `server.command_metrics()` 能看到 `PING`、`STATS`、`HEALTH`、`QUIT` 四个命令桶。
  - 过长请求在进入 `CommandProcessor` 前被 TCP 层拒绝，不污染命令指标。
  - 结构化日志中包含 `avg_latency_ns` 和 `command_breakdown`。

## 06-command-breakdown-latency-smoke.png

- 命令：真实启动 `minikv_server.exe`，再用真实 `minikv_client.exe` 通过重定向 stdin 输入命令。
- client 输入流程：

```text
PING
BADCOMMAND
STATS
HEALTH
QUIT
```

- 关键结果：
  - client 收到 `PONG`。
  - `BADCOMMAND` 返回 `ERR unknown command`。
  - `STATS` 返回 `total_commands=2 successful_commands=1 error_commands=1`。
  - `STATS` 包含 `total_latency_ns`、`avg_latency_ns`、`max_latency_ns`。
  - `STATS` 包含 `command_breakdown=PING:1/1/0/...` 和 `UNKNOWN:1/0/1/...`。
  - `HEALTH` 返回 `total_commands=3 successful_commands=2 error_commands=1`，并能看到 `STATS:1/1/0/...`。
  - server 周期性 `event=server_metrics` 日志包含 `command_breakdown` 和 latency 字段。
  - 验证结束后停止本轮启动的 server 进程。

调试备注：

```text
smoke 输入文件继续使用 UTF-8 无 BOM 写入，避免第一条命令前带 BOM 导致 PING 被误识别为未知命令。
readiness probe 只建立连接，不发送命令，所以可能增加 total_connections，但不会影响 command metrics。
```
