# mini-kv v146 命令归档说明

本版归档对象是 `SHARDJSON` historical fallback hardening。v146 的重点不是新增分片能力，而是把 Node 已消费的 mini-kv v145 证据冻结为 `fixtures/release/shard-readiness-v145.json`，避免后续历史验证继续读取滚动 `fixtures/release/shard-readiness.json`。

截图使用 Playwright MCP 生成；由于 MCP 禁止直接打开 `file://`，归档时临时启动本地只读 HTTP 页面服务承载日志，截图完成后关闭。

## 截图

- `01-cmake-configure.png`
  - 命令：`cmake -S D:\C\mini-kv -B D:\C\mini-kv\cmake-build-v146 -G Ninja ...`
  - 结果：CMake 配置和生成成功。
  - 意义：证明 v146 的 historical fallback hardening 源码和新增冻结 fixture 可被当前工具链识别。

- `02-target-build-ctest.png`
  - 命令：构建 `minikv_shard_readiness_tests`、`minikv_command_tests`，随后定向运行 `ctest -R shard_readiness_tests|command_tests`。
  - 结果：2/2 通过。
  - 意义：证明 current v146 fixture 等于 runtime 输出，v144/v145 历史 fixture 保留，且 `SHARDJSON` 仍然是只读、非 WAL、拒绝多余参数的命令。

- `03-full-build-ctest.png`
  - 命令：全量 `cmake --build` 和全量 `ctest --output-on-failure`。
  - 结果：71/71 通过。
  - 意义：证明 v146 没有破坏既有 WAL、snapshot、runtime evidence、credential resolver non-participation、TCP/RESP 和 restore evidence 回归链。

- `04-tcp-shardjson-historical-fallback-smoke.png`
  - 命令：临时启动 `minikv_server.exe 6416 127.0.0.1`，用真实 `minikv_client.exe` 读取 `HEALTH`、`COMMANDSJSON`、`EXPLAINJSON SHARDJSON`、`CHECKJSON SHARDJSON extra`、`SHARDJSON`、`QUIT`。
  - 结果：真实 TCP 输出包含 `releaseVersion=v146`、`status=historical-fallback-hardened-read-only`、`historicalFallback.previousConsumedFixturePath=fixtures/release/shard-readiness-v145.json`、`rollingCurrentUsedForHistoricalBaseline=false` 和 `evidenceDigest=fnv1a64:6847d87decb76fcb`。服务端验证后已停止，`6416` 没有 LISTENING 残留。
  - 意义：证明 v146 证据不仅在 fixture 中存在，真实命令路径也能读到历史 fallback 边界。

## 边界

v146 仍不创建分片目录，不安装 active router，不启动多进程，不写存储，不执行 `LOAD`、`COMPACT`、`RESTORE`、`SETNXEX`，也不让 mini-kv 成为订单或审计权威。`shard-readiness-v145.json` 是冻结历史证据，专门避免旧版本验证误读滚动 current。
