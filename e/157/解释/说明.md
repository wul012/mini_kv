# mini-kv v157 命令归档说明

本轮先只读确认 Node 当前计划为 `docs/plans3/v402-post-runtime-execution-approval-input-template-validator-roadmap.md`。该计划只发布三个 runtime approval input 模板，不创建真实 canonical input，不启动 Java / mini-kv。mini-kv v157 因此只做只读模板回显与 v156 证据冻结。

## 01-cmake-configure-generate-fixture.png

命令：

```powershell
cmake -S . -B cmake-build-v157 -G Ninja
cmake --build cmake-build-v157 --target minikv_cli
minikv_cli.exe stdin: SHARDJSON
```

结果：CMake 配置成功，`minikv_cli.exe` 构建成功，当前 `fixtures/release/shard-readiness.json` 由真实 CLI 路径生成。生成后的 `releaseVersion=v157`，`status=runtime-execution-approval-input-template-validator-echo-read-only`，`evidenceDigest=fnv1a64:1d8cdf7f597f837e`。

意义：证明 current fixture 不是手写孤本，而是由当前 C++ 实现输出；v157 模板回显仍保持 `executionAllowed=false`。

## 02-target-build-ctest.png

命令：

```powershell
cmake --build cmake-build-v157 --target minikv_shard_readiness_tests minikv_command_tests
ctest --test-dir cmake-build-v157 -R "^(command_tests|shard_readiness_tests)$" --output-on-failure
```

结果：定向构建成功，`command_tests` 与 `shard_readiness_tests` 2/2 通过。

意义：证明 `SHARDJSON` 的 formatter、命令入口、v156 冻结 fixture、v157 `runtimeExecutionApprovalInputTemplateValidatorEcho`、no router、no write、no execution 字段被重点测试覆盖。

## 04-tcp-smoke-cleanup.png

命令：

```powershell
minikv_server.exe 6428 127.0.0.1
minikv_client.exe 127.0.0.1 6428
```

客户端输入：

```text
SHARDJSON
GET v157:template-echo:probe
QUIT
```

结果：真实 TCP 路径读到 v157 `SHARDJSON`，包含 `runtimeExecutionApprovalInputTemplateValidatorEcho`，`templateCopiedToCanonicalInput=false`，`routerActivationAllowed=false`，`writeRoutingAllowed=false`，`executionAllowed=false`。`GET v157:template-echo:probe` 返回 `(nil)`。声明的 approval port `6424` 全程未监听；临时 smoke port `6428` 验证后已停止并确认不再监听。

意义：证明本版只读证据可以通过真实服务读取，但不会把模板变成 canonical approval input，不会启动声明端口，不会写 key，不会启用 router/write/execution。
