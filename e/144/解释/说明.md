# mini-kv v144 命令归档说明

本版归档对象是 mini-kv v144 的只读分片就绪雏形证据。截图使用 Playwright MCP 生成；由于 MCP 禁止直接打开 `file://` 页面，归档时临时启动了只读本地 HTTP 页面服务来承载命令日志，截图完成后按清理门禁关闭。

## 截图

- `01-cmake-configure.png`
  - 命令：`cmake -S D:\C\mini-kv -B D:\C\mini-kv\cmake-build-v144 -G Ninja ...`
  - 结果：CMake 配置和生成成功，构建目录为 `cmake-build-v144`。
  - 意义：证明 v144 的新增源文件、测试目标和 CMake 目标可以被当前 CLion/MinGW/Ninja 工具链识别。

- `02-target-build-ctest.png`
  - 命令：`cmake --build ... --target minikv_shard_readiness_tests minikv_command_tests`，随后 `ctest -R shard_readiness_tests|command_tests --output-on-failure`。
  - 结果：目标构建成功；`command_tests` 和 `shard_readiness_tests` 共 2/2 通过。
  - 意义：证明 `SHARDJSON` 命令注册、命令目录曝光、fixture 等值校验和分片就绪 JSON 字段都被定向测试覆盖。

- `03-full-build-ctest.png`
  - 命令：`cmake --build D:\C\mini-kv\cmake-build-v144`，随后 `ctest --test-dir D:\C\mini-kv\cmake-build-v144 --output-on-failure`。
  - 结果：全量构建成功；CTest 71/71 通过。
  - 意义：证明 v144 没有破坏既有 WAL、snapshot、runtime evidence、credential resolver non-participation、TCP/RESP、restore evidence 等历史回归链。

- `04-tcp-shardjson-smoke.png`
  - 命令：临时启动 `minikv_server.exe 6414 127.0.0.1`，用 `minikv_client.exe` 发送 `HEALTH`、`INFOJSON`、`STATSJSON`、`SHARDJSON`、`QUIT`。
  - 结果：客户端拿到完整 `SHARDJSON`，其中 `contract=shard-readiness.v1`、`readOnly=true`、`executionAllowed=false`、`shardEnabled=false`、`shardCount=1`、`slotCount=16`、`evidenceDigest=fnv1a64:22d3c4815a440804`。服务端验证后已停止，`6414` 没有 LISTENING 残留。
  - 意义：证明新证据不是只在单元测试中存在，真实 TCP 命令路径也能读到同一份只读分片就绪契约。

## 边界

v144 不创建分片目录，不启动多进程，不写存储，不执行 `LOAD`、`COMPACT`、`RESTORE`、`SETNXEX`，也不让 mini-kv 成为订单或审计权威。它只给 Node v370 和后续 Java echo 提供 `shard-readiness.v1` 的只读消费面。
