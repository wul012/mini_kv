# mini-kv v152 命令归档说明

本版归档对象是 `SHARDJSON` declared operator lifecycle evidence。v152 先把 v151 的 `operatorServiceLifecycleTemplate` 证据冻结成 `fixtures/release/shard-readiness-v151.json`，再让 rolling current 输出新增 `operatorServiceLifecycleEvidence`。它声明本地 operator handle、启动命令形状、operator-approved loopback port handle、GET-only `SHARDJSON` smoke target、fail-closed 行为和 cleanup 责任。

这个声明仍然不是 runtime gate approval。`runtimeGateApproved=false`，所以 mini-kv 不启动服务作为证据动作，不运行 runtime probe，不打开 live read，不安装 active router，不创建分片目录，不写入存储，也不执行 `LOAD`、`COMPACT`、`RESTORE` 或 `SETNXEX`。截图使用 Playwright MCP 生成；归档时临时启动本地只读 HTTP 页面服务承载日志渲染页，截图完成后已关闭该服务。

## 截图

  - 命令：`cmake -S . -B cmake-build-v152 -G Ninja`
  - 结果：CMake 配置和生成成功。
  - 意义：证明 v152 构建树能识别新增的 declared lifecycle evidence 代码、测试和 v151 frozen fixture。

- `02-target-build-ctest.png`
  - 命令：构建 `minikv_shard_readiness_tests`、`minikv_command_tests`，随后运行 `ctest -R '^(command_tests|shard_readiness_tests)$'`。
  - 结果：2/2 通过。
  - 意义：证明 current v152 fixture 等于 runtime 输出，v151 frozen fixture 保留，`operatorServiceLifecycleTemplateFreeze` 和 `operatorServiceLifecycleEvidence` 均保持 no runtime probe、no live read、no router、no write、no execution。

- `03-full-build-ctest.png`
  - 命令：全量 `cmake --build cmake-build-v152` 和全量 `ctest --output-on-failure`。
  - 结果：71/71 通过。
  - 意义：证明 v152 没有破坏既有存储、WAL、snapshot、restore、TCP/RESP、runtime evidence 和 credential resolver non-participation 证据链。

- `04-tcp-shardjson-declared-lifecycle-evidence-smoke.png`
  - 命令：临时启动 `minikv_server.exe 6422 127.0.0.1`，用真实 `minikv_client.exe` 读取 `HEALTH`、`COMMANDSJSON`、`EXPLAINJSON SHARDJSON`、`CHECKJSON SHARDJSON extra`、`SHARDJSON`、`QUIT`。
  - 结果：真实 TCP 输出包含 `releaseVersion=v152`、`status=declared-operator-lifecycle-no-runtime-read-only`、`previousConsumedFixturePath=fixtures/release/shard-readiness-v151.json`、`operatorServiceLifecycleTemplateFreeze.frozenReleaseVersion=v151`、`operatorServiceLifecycleEvidence.evidenceMode=declared-lifecycle-no-runtime`、`operatorOwnedServiceLifecycleDeclared=true`、`serviceOwnerDeclared=true`、`startupCommandDeclared=true`、`portListDeclared=true`、`getOnlySmokeTargetDeclared=true`、`failClosedBehaviorDeclared=true`、`cleanupResponsibilityDeclared=true`，同时包含 `runtimeGateApproved=false`、`startsServices=false`、`runtimeProbeAllowed=false`、`liveReadAllowed=false`、`routerActivationAllowed=false`、`writeRoutingAllowed=false`、`executionAllowed=false`、`writeCommandsAllowed=false` 和 `activeRouterInstalled=false`。服务端验证后已停止，`6422` 没有 LISTENING 残留。
  - 意义：证明 v152 通过真实命令路径暴露 operator lifecycle 声明证据，但没有把声明误升级成 runtime 授权、服务启动、runtime probe、router 或写入能力。

## 边界

v152 仍是只读 evidence release。`operatorServiceLifecycleEvidence` 的含义是“operator lifecycle 所需的信息形状已经声明”，不是“mini-kv 可以开始 live read”。后续若要进入 runtime live-read gate，仍必须有单独批准、具体 loopback port、GET-only smoke 命令和 cleanup proof。mini-kv 不读取或修改 Node 归档，不启动 Java/Node，不成为 Java order authority，也不承担 audit authority。
