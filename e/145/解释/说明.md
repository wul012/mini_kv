# mini-kv v145 命令归档说明

本版归档对象是 `SHARDJSON` 只读分片就绪 hardening。它不改变 Node v370-v373 已归档证据链，只产出 mini-kv v145 的新只读证据，供后续 Node 版本再消费。截图使用 Playwright MCP 生成；由于 MCP 禁止直接打开 `file://`，归档时临时启动本地只读 HTTP 页面服务承载日志，截图完成后关闭。

## 截图

- `01-cmake-configure.png`
  - 命令：`cmake -S D:\C\mini-kv -B D:\C\mini-kv\cmake-build-v145 -G Ninja ...`
  - 结果：CMake 配置和生成成功。
  - 意义：证明 v145 的 `SHARDJSON` hardening 源码和新增 fixture 能被当前工具链识别。

- `02-target-build-ctest.png`
  - 命令：构建 `minikv_shard_readiness_tests`、`minikv_command_tests`，随后定向运行 `ctest -R shard_readiness_tests|command_tests`。
  - 结果：2/2 通过。
  - 意义：证明 `SHARDJSON` 的命令目录、fixture parity、历史 v144 fixture 保留、只读边界字段和 usage 拒绝语义都有测试覆盖。

- `03-full-build-ctest.png`
  - 命令：全量 `cmake --build` 和全量 `ctest --output-on-failure`。
  - 结果：71/71 通过。
  - 意义：证明 v145 没有破坏既有 WAL、snapshot、runtime evidence、credential resolver non-participation、TCP/RESP 和 restore evidence 回归链。

- `04-tcp-shardjson-hardening-smoke.png`
  - 命令：临时启动 `minikv_server.exe 6415 127.0.0.1`，用真实 `minikv_client.exe` 读取 `HEALTH`、`COMMANDSJSON`、`EXPLAINJSON SHARDJSON`、`CHECKJSON SHARDJSON extra`、`SHARDJSON`、`QUIT`。
  - 结果：真实 TCP 输出包含 `releaseVersion=v145`、`status=hardened-read-only`、`commandCatalog`、`fixtureParity`、`archiveCompatibility`、`readOnlyBoundaryFields` 和 `evidenceDigest=fnv1a64:ebe4c7e1a2704482`。服务端验证后已停止，`6415` 没有 LISTENING 残留。
  - 意义：证明 hardening 字段不只存在于 fixture 和单元测试里，真实命令路径也能读到，并且多余参数仍被 `CHECKJSON` 标为 parser 不允许。

## 边界

v145 仍不创建分片目录，不安装 active router，不启动多进程，不写存储，不执行 `LOAD`、`COMPACT`、`RESTORE`、`SETNXEX`，也不让 mini-kv 成为订单或审计权威。`fixtures/release/shard-readiness-v144.json` 只用于保留 v144 历史证据，不代表 Node v370-v373 归档链被重写。
