# mini-kv v151 命令归档说明

本版归档对象是 `SHARDJSON` operator service lifecycle template evidence。v151 不声明 operator owner，不给出实际启动命令，不给出端口清单，不声明 GET-only smoke target，不启动长期服务，不运行 runtime probe，不安装 active router，不创建分片目录，不写入存储，也不执行 `LOAD`、`COMPACT`、`RESTORE` 或 `SETNXEX`。

v151 做的事情很窄：把 v150 `liveReadGatePlan` 证据冻结成 `fixtures/release/shard-readiness-v150.json`，再在 rolling current 中新增 `operatorServiceLifecycleTemplate`。这个字段只是把 Node v385 后续要求的 operator-owned service lifecycle evidence 结构化成模板：service owner、startup command、port list、GET-only smoke target、fail-closed behavior、cleanup responsibility 都仍需后续独立提供。

截图使用 Playwright MCP 生成。由于 MCP 不允许直接打开 `file://` 页面，本轮临时启动了一个只读本地 HTTP 页面服务承载日志渲染页；截图完成后已关闭该服务。

## 截图

- `01-cmake-configure.png`
  - 命令：`cmake -S . -B cmake-build-v151 -G Ninja`
  - 结果：CMake 配置和生成成功。
  - 意义：证明 v151 构建树可以识别新增的 operator service lifecycle template 字段和版本化 v150 fixture。
- `02-target-build-ctest.png`
  - 命令：构建 `minikv_shard_readiness_tests`、`minikv_command_tests`，随后运行 `ctest -R '^(command_tests|shard_readiness_tests)$'`。
  - 结果：2/2 通过。
  - 意义：证明 current v151 fixture 等于 runtime 输出，v150 frozen fixture 被保留，`liveReadGatePlanFreeze` 和 `operatorServiceLifecycleTemplate` 都保持 no runtime probe、no service start、no router、no write、no execution。
- `03-full-build-ctest.png`
  - 命令：全量 `cmake --build cmake-build-v151` 和全量 `ctest --output-on-failure`。
  - 结果：71/71 通过。
  - 意义：证明 v151 没有破坏既有存储、WAL、snapshot、restore、TCP/RESP、runtime evidence 和 credential resolver non-participation 证据链。
- `04-tcp-shardjson-service-lifecycle-template-smoke.png`
  - 命令：临时启动 `minikv_server.exe 6421 127.0.0.1`，用真实 `minikv_client.exe` 读取 `HEALTH`、`COMMANDSJSON`、`EXPLAINJSON SHARDJSON`、`CHECKJSON SHARDJSON extra`、`SHARDJSON`、`QUIT`。
  - 结果：真实 TCP 输出包含 `releaseVersion=v151`、`status=operator-service-lifecycle-template-read-only`、`previousConsumedFixturePath=fixtures/release/shard-readiness-v150.json`、`liveReadGatePlanFreeze.frozenReleaseVersion=v150`、`operatorServiceLifecycleTemplate.evidenceMode=template-only-no-runtime`、`operatorOwnedServiceLifecycleRequired=true`、`serviceOwnerDeclared=false`、`startupCommandDeclared=false`、`portListDeclared=false`、`getOnlySmokeTargetDeclared=false`、`cleanupResponsibilityDeclared=false`、`startsServices=false`、`runtimeProbeAllowed=false`、`liveReadAllowed=false`、`routerActivationAllowed=false`、`writeRoutingAllowed=false`、`executionAllowed=false`、`writeCommandsAllowed=false` 和 `activeRouterInstalled=false`。服务端验证后已停止，`6421` 没有 LISTENING 残留。
  - 意义：证明 v151 通过真实命令路径暴露 operator lifecycle 模板，但没有把模板误变成 operator 授权、服务启动或 runtime probe。

## 边界

v151 仍是只读 evidence release。`operatorServiceLifecycleTemplate` 只说明后续需要什么，不代表这些证据已经存在；后续若要 live read，仍必须由 operator 明确 service owner、启动命令、端口清单、GET-only smoke target、fail-closed 行为和 cleanup 责任。mini-kv 不读取或修改 Node 归档，不启动 Java/Node，不成为 Java order authority，也不承担 audit authority。
