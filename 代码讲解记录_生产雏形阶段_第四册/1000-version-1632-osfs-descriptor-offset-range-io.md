# v1632 代码讲解：从“句柄包装整文件”推进到可观察的文件描述符位移与区间 I/O

## 1. 版本目标与“不是什么”

v1631 已经把用户表、MFD 和 UFD 真正写进磁盘镜像，但文件描述符仍然很薄：`Handle` 只保存文件名以及可读、可写标志，`READ` 每次都返回整个文件，`WRITE` 每次都替换整个文件。这个行为可以演示权限检查，却不能解释操作系统为什么要维护打开文件表，也无法展示连续读、连续写、追加和当前位置这些核心概念。

v1632 的目标是让每个打开句柄拥有独立的读位置和写位置。读取从当前读位置开始，返回指定长度后推进；写入从当前写位置覆盖或扩展文件，成功后推进。`SEEK` 改变句柄位置，`TELL` 显示位置。`OPEN` 模式决定初始状态：`r` 从文件头读，`w` 先截断再从头写，`rw` 保留原内容并从头进行定位读写，`a` 把写位置放到文件末尾。

这不是把 OSFS 改造成完整 POSIX 文件系统。本版不实现共享系统级 open-file description、不处理多进程并发、不做稀疏文件语义规范、不增加间接块，也不把 fd 暴露给 mini-kv 的 TCP/RESP 路由。它仍是单进程课程设计 shell，但内部机制足以清楚回答“fd 中保存了什么”和“read/write 如何改变当前位置”。

## 2. 命令入口和句柄结构

命令入口仍在 `src/osfs_command_processor.cpp`。`CommandProcessor::Handle` 现在包含 `name`、`readable`、`writable`、`read_offset` 和 `write_offset`。偏移属于句柄，而不是 inode：同一个文件被打开两次会得到两个 fd，每个 fd 可以独立推进。用户重新登录时句柄表清空，防止前一身份的打开状态泄露到后一身份。

`OPEN name r` 创建只读句柄，两个偏移从 0 开始；`OPEN name w` 先验证写权限，再调用 `truncate_file` 释放旧数据块并把文件长度设为 0；`OPEN name rw` 不截断，读写位置都从 0 开始；`OPEN name a` 读取 stat 中的文件长度，把写位置初始化为 EOF。无论模式如何，权限都在创建句柄之前检查，因此不能通过 `OPEN w` 的截断副作用绕过写权限。

`READ fd [length]` 允许省略长度。指定长度时，只读取“长度、剩余文件字节”两者中的较小值；省略时读取到 EOF。命令层调用 `FileSystem::read_file_range`，成功后用实际返回字节数推进 `read_offset`。到达 EOF 返回 `(eof)`，位置保持不变。`WRITE fd text` 调用 `write_file_range`，只有底层成功后才推进 `write_offset`；空间不足、权限拒绝或超过直接块上限时，句柄位置不会虚假前移。

`SEEK fd offset` 对可读句柄设置读位置，对可写句柄设置写位置，`rw` 句柄会同时设置两者。允许把写位置放在当前 EOF 之后，后续写入时中间区域由零填充块表达；如果目标超出八个直接块可表示的最大文件长度，真正写入会在分配前失败。`TELL` 输出两个位置，使演示者不需要猜测内部状态，也使测试能够检查位置变化而不是只检查最终文件内容。

## 3. 区间读取如何跨数据块

底层实现在 `src/osfs_file_io.cpp`。公开接口 `read_file_range(name, offset, optional_length, uid, error)` 首先沿 v1631 的用户表和 UFD 路径找到文件 inode，然后进行 owner/group/other 读权限判断。它不会因为调用者知道 `alice/file` 这样的限定名字就跳过权限。

读取长度先限制在文件剩余长度内。如果 offset 已经大于等于文件长度，函数更新 atime 并返回空字符串。否则循环计算 `direct_index = file_offset / block_size` 和 `within_block = file_offset % block_size`：前者选择 inode 的第几个直接块，后者确定块内起点。每轮最多复制到当前块末尾，之后进入下一个直接块，直到满足长度。这样从 offset 500 读取 40 字节时，在 512 字节块大小下会先取第一个块末尾 12 字节，再从第二块取 28 字节，测试明确覆盖了这个边界。

原来的 `read_file` 没有被删掉，而是成为 `read_file_range(name, 0, nullopt, ...)` 的兼容包装。这避免维护两套读取算法，也保证直接 API、命令 shell 和既有测试看到一致的权限、atime 和数据块行为。

## 4. 区间写入、扩展和失败边界

`write_file_range` 同样先解析用户/UFD/inode并检查写权限。空字符串写入被视为成功但不扩展文件。对非空内容，函数先计算 `offset + size`，并与 `max_direct_blocks * block_size` 比较；计算采用“先检查剩余容量再相加”的形式，避免无符号溢出把超大请求误判为小文件。

目标文件大小是原大小和写入末端中的较大值。根据目标大小计算需要的数据块数量，再统计 inode 当前已经分配的直接块数量。如果需要新增块，先比较 block bitmap 中的空闲块数；空间不足直接返回，原 inode、位图、数据块和 fd 位置都不改变。空间充足时才逐个分配缺少的块，新块先清零，因此从 EOF 之后开始写时中间空洞不会暴露旧磁盘字节。

实际复制与区间读取使用相同的块索引计算。已有块先读出，只有目标区间被覆盖，其余字节保持不变；新块从全零内容开始。一次写入跨块时，循环会在块末尾切分，不依赖数据块在磁盘上的物理连续性。完成后 inode size 只增不减，mtime 更新，block bitmap 和超级块空闲块计数一起落盘。

`truncate_file` 是独立操作，只由 `OPEN w` 调用。它在权限检查后释放 inode 的全部直接块、清零块内容、把长度设为 0，并更新 bitmap、mtime 和超级块计数。把截断从 `open_file` 的字符串逻辑下沉到文件系统层，可以保证任何未来调用者都遵循相同的空间回收规则。

原有 `write_file` 仍提供整文件替换语义，供测试和低层 API 使用。它保留“先确认新内容所需空间，再释放旧块”的保护，空间不足时旧内容仍然可读。命令层的新连续写不再调用这个接口，而是使用 range 写，避免每次追加都重新分配整个文件。

## 5. 测试为何能证明不是字符串拼接

新增 `test_descriptor_offsets_and_range_io` 先以 `w` 模式打开空文件，连续写入 `hello-` 和 `world`。两次响应分别报告 offset 6 和 11，`TELL` 也必须显示 write_offset 11。关闭后以新的只读句柄打开，按 5 字节、1 字节、剩余字节三次读取，必须依次得到 `hello`、`-`、`world`，并在下一次读取返回 EOF。这组断言保护“按实际返回量推进”，而不是每次重复整个文件。

随后测试用 `rw` 打开同一文件，`SEEK 6` 后写入 `OSFS!`，再回到 0 读取，结果必须是 `hello-OSFS!`。这证明写入是在指定位置覆盖已有字节，不是简单尾部拼接。再以 `a` 模式打开，`TELL` 必须直接显示 write_offset 11，追加 `-append` 后最终内容为 `hello-OSFS!-append`。关闭并重新打开整个磁盘镜像后仍得到相同内容，说明变更落在数据块和 inode 中，而非句柄内的临时字符串。

测试还创建一个 700 字节文件，从 offset 500 读取 40 字节，再从 508 写入 `boundary-write` 并读回。这个区间横跨 512 字节块边界，能够抓住只更新单块、块内偏移计算错误或第二块未分配等实现问题。既有空间不足测试继续保护整文件替换失败不破坏旧内容，v1631 的认证、目录隔离和权限测试则保证 range I/O 没有绕过用户边界。

真实 `osfs_cli_smoke` 也改为连续写和分段读：Alice 连续写出 `course-` 与 `design`，关闭后重新打开并分别读取六字节、一字节和剩余部分，最终 `TELL` 显示 read_offset 13。脚本之后继续切换 Bob，验证目录为空且访问 Alice 的 `0600` 文件失败，再由 root 查看 Alice UFD。于是同一条真实演示同时覆盖认证、二级目录、权限和 fd 位移。

## 6. 维护性和下一步边界

v1632 没有把偏移算法塞回已经拆开的生命周期或目录模块。`osfs_file_io.cpp` 集中处理数据块范围，`osfs_command_processor.cpp` 只保存会话句柄和解析命令。最大 OSFS 文件仍处于数百行规模，没有重新形成 v1629 的六百行单体。公开头文件新增的 range API 使用 `std::optional<size_t>` 表达“读到 EOF”，比用特殊大整数更清楚。

本版完成后，brief 中两个功能硬缺口已经关闭：v1631 解决 MFD/UFD 和持久化用户登录，v1632 解决 fd 读写位置。下一阶段才允许重做报告、磁盘布局图、演示 transcript 和需求-证据矩阵。最终材料必须把 `w/rw/a` 语义、MFD/UFD、用户表和双位图写入设计章节，并明确八个直接块、单进程句柄和教学密码哈希仍是边界。

一句话总结：v1632 让 OSFS 的 open/read/write/close 从命令外形升级为真正可解释的打开文件表机制，同时仍保持 KV/WAL/TCP 主线隔离，为最终课程交付提供了能被真实 demo 和跨块测试复核的执行证据。
