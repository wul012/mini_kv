# v1630 OSFS 课程设计交付收尾讲解

本版的目标不是继续给 OSFS 增加新命令，也不是把 mini-kv 的 KV、WAL、TCP 主线改造成文件系统。v1629 已经完成了真正的运行时能力：`minikv_osfs` 可以格式化二进制磁盘镜像，写入超级块、块位图、inode 表、根目录和数据块，并通过 `LOGIN`、`CREATE`、`OPEN`、`WRITE`、`READ`、`CHMOD`、`DIR`、`STAT` 等命令完成课程演示。v1630 做的是另一类工作：把这些能力整理成一套可以提交、可以打印、可以答辩、可以复现的课程设计交付包。换句话说，v1629 解决“系统能不能跑”，v1630 解决“老师能不能看懂、学生能不能讲清、平台能不能提交”。

本版首先重新读取了课程设计 PDF 的硬要求。PDF 中“实验二 Linux 二级文件系统”要求用二进制文件模拟文件系统，保存用户信息、inode 信息、超级块信息和文件信息，实现 `dir`、`create`、`delete`、`open`、`close`、`read`、`write`、`login` 等命令，并且支持读写保护。这个要求和 v1629 的 OSFS 很贴合，因此 v1630 没有必要继续扩功能。继续加 `mkdir`、多级目录、group 权限或间接块当然能让系统更完整，但在课设提交节点前，最有价值的工作是把现有功能讲清楚、跑稳、归档好。这里体现的工程判断是：功能够覆盖验收时，继续堆功能的收益开始下降；报告、截图、演示脚本和提交清单反而会直接影响最终交付质量。

本版新增的核心文件是 `tools/build_osfs_coursework_report.py`。它不是运行时模块，不参与 `minikv_osfs` 的命令执行，也不改变 `FileSystem`、`VirtualDisk`、`CommandProcessor` 的行为。它的职责是生成课程报告和三张结构图：第一张是 OSFS 总体结构图，说明用户命令、命令处理层、文件系统层、虚拟磁盘层和原 mini-kv 主线之间的边界；第二张是虚拟磁盘布局图，展示 Block 0 超级块、Block 1 位图、inode table、root directory 和数据块；第三张是 `WRITE` 命令处理流程图，重点标出 fd 检查、权限判断、空间预检查、数据块分配以及失败不破坏旧内容的边界。这三张图不是装饰，而是把课程设计中最容易讲散的部分压成可视结构。答辩时先看图，再讲代码，学生更容易把“文件系统内部结构”讲成一条清楚的线。

报告生成脚本使用 `python-docx` 创建 `操作系统课程设计报告-OSFS-v1630.docx`，再通过 LibreOffice 导出 PDF。报告正文没有写成泛泛的“系统具有重要意义”，而是围绕具体课设要求组织：选题来源、需求映射、总体结构、磁盘布局、核心操作流程、权限模型、模块说明、测试与演示、运行复现、不足与改进。这样的结构有两个好处。第一，它能直接对应老师给的评分点，尤其是实验二要求中的“二进制文件模拟磁盘”“inode 信息”“超级块信息”“dir 列字段”“login 用户交互”和“读写保护”。第二，它不会把 OSFS 说成生产级文件系统。报告明确写出当前只支持根目录、直接块、owner/other 权限，没有多级目录、group 权限和间接块。这个限制说明不是自我削弱，而是让报告更可信：一个课程设计系统如果声称自己完整复刻 EXT2，反而很容易被追问无法回答的细节。

交付目录是 `课程设计交付/v1630-osfs-final/`。里面的 `README.md` 说明每个文件的作用；`演示步骤.md` 给出课堂演示顺序；`提交清单.md` 把纸质报告、成绩考核表、电子版报告、源码路径和演示材料列出来；`osfs-demo-script.txt` 是实际传给 `minikv_osfs --script` 的命令脚本；`osfs-demo-transcript.txt` 是本版真实运行输出；`images/` 放报告插图。这个目录的价值在于把“代码仓库”变成“课程提交包”。如果只给老师一个 C++ 仓库，对方还要自己判断哪些文件是课设入口、怎么构建、怎么演示、报告在哪里、截图是不是来自真实运行。v1630 把这些问题提前回答掉。

演示脚本的输入也经过了选择。它不是简单跑 `HELP`，而是按课设验收顺序串起来：先 `WHOAMI` 证明默认 root 用户，再 `DIR` 证明空目录；随后 `CREATE report` 创建文件，`OPEN report rw` 得到 fd，`WRITE 3 course design storage layer` 写入正文，`READ 3` 读回；接着 `CHMOD report 0600` 修改保护码，`LOGIN guest` 切到普通用户，再 `OPEN report r` 得到 `ERR permission denied`；最后 `DIR` 输出 `name inode physical mode owner size`，`STAT report` 输出 inode、physical、mode、owner、size 和时间戳。这个顺序能自然覆盖课程要求里的命令、目录列字段和读写保护。尤其是权限拒绝那一步，它证明保护码不是报告里的静态字段，而是真实参与了命令执行。

本版的归档证据放在 `f/1630/`。`01-course-requirements.png` 证明我们确实读取了课程 PDF 的提交要求和实验二要求；`02-build-focused-tests.png` 证明 `minikv_osfs`、`minikv_osfs_tests` 和 OSFS focused CTest 通过；`03-osfs-course-demo.png` 是真实演示输出；`04-report-render-check.png` 证明 DOCX 转 PDF 并渲染成页面 PNG 后做了视觉检查；`05-full-ctest-summary.png` 证明全量本地 CTest 344/344 通过。这些截图不是为了堆材料，而是把每个关键判断都留下证据：要求是否匹配、代码是否能构建、演示是否真实、报告是否可打开、旧测试是否没被破坏。

报告渲染闭环是本版比较重要的一点。Word 报告如果只生成 `.docx` 文件，还不能保证页面不会出现图片压缩、表格裁切、中文字体丢失或分页混乱。v1630 用 LibreOffice 把 DOCX 导出 PDF，再用 Poppler 把 PDF 渲染成 8 张 PNG，并人工抽查封面、结构图页、流程图页、演示截图页、测试证据页和结尾页。检查结果显示封面表格清楚，结构图和流程图没有被裁切，演示截图和全量 CTest 截图已经换成 v1630 自己的证据，结尾页没有重叠。LibreOffice 和 Poppler 在本机中文用户名路径下输出了编码警告，但最终 DOCX、PDF 和页面 PNG 都成功生成，因此这是环境警告，不是交付失败。

从边界上看，v1630 没有给 mini-kv 增加新的网络入口，没有启动 `minikv_server`，没有打开 Node 或 Java 的 managed-audit 链路，也没有更改 WAL、snapshot、restore、TCP/RESP 的行为。OSFS 仍然是独立课设入口，原项目主线仍然是 C++ KV 存储系统。全量 CTest 344/344 通过，说明这次文档和交付包整理没有污染已有测试面。这一点对长期维护很重要：课程设计材料可以增强项目表达力，但不能因为报告要好看就把运行时边界弄乱。

本版也留下了后续路线。若继续做 OSFS，比较自然的是先加多级目录，也就是 `mkdir`、`cd`、`pwd` 和目录 inode 的递归管理；再加 group 权限和间接块；最后可以考虑 `fsck` 类检查命令，用来验证位图、inode 表和目录项是否一致。但这些都应该是新的功能版本，而不是夹在交付收尾版里。v1630 的价值在于停在一个合适的位置：功能足以覆盖课设主体，报告足以支撑提交，证据足以说明真实性，边界也没有被破坏。对后续 Node/Java/mini-kv 的松耦合融合来说，这一版提供的不是新接口，而是一个清楚的例子：学习型扩展可以和生产型证据链并存，只要入口独立、材料完整、测试闭环、边界讲明白。
