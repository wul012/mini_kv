# mini-kv v1659 代码讲解：把“优雅”变成会失败的命名债务合同

## 一、本版目标、角色定位与明确边界

v1659 是 `elegance-hotspot-program-minikv.md` 的 E-M1，它不直接替换那些已经长到一百多个字符的文件名，也不试图在一个版本里把历史命名债务清空。本版承担的角色是先建立测量仪器：把什么叫“超过命名预算”、当前到底有多少存量、哪些名字属于公共头文件表面，以及以后什么变化必须被拒绝，全部转成可以由 CTest 自动判定的合同。只有先固定口径，后面的 top-5 改名才能证明是在还债，而不是把旧名字换成另一批同样糟糕的新名字。

这不是新的 KV 命令，不是新的控制面入口，也不是对 receipt、WAL、Snapshot、RESP 或 OSFS 的功能扩展。它不会修改 `SET`、`GET`、`CHECKJSON`、`SMOKEJSON` 或 `SHARDJSON` 的执行路径，不会安装 active router，不会获得 write routing、网络访问、凭据读取、恢复、压缩、部署或回滚能力。它也不把“命名扫描通过”包装成生产数据库已经完善。v1658 的外部 Stage-1 PASS 仍然是前一阶段结论，v1659 只是该结论之后的一次有界维护工作。

本版还特别避免触碰三类稳定表面。第一，`fixtures/release/*.json` 和 `a/` 到 `f/` 的历史路径是冻结证据，不能为了目录好看而搬动。第二，CLI 命令名、RESP/JSON 键和公共函数签名可能被 Node、Java、脚本或外部使用者消费，不能只凭“名字太长”就改掉。第三，公共头文件中的长名字虽然会被 census 看见，但 E-M2 只能选择没有外部合同钉住的内部实现文件；发现债务不等于获得破坏兼容性的许可。

## 二、为什么必须先做 census，而不是马上批量改名

过去的长命名并不是偶发的一个文件，而是治理证据随版本叠加后形成的系统性模式。用一次 `rg` 或 PowerShell 排序可以看到最坏样本，但这种临时命令没有稳定语义：有人可能按完整路径长度统计，有人只数 basename；有人把扩展名算进去，有人不算；有人用正则扫描所有 `.cpp` 局部变量，却把结果称为“公共 API”。如果口径不固定，版本 A 声称减少五个，版本 B 换一个脚本又可能声称增加几十个，数字无法复核，更不能成为 ratchet。

因此 v1659 先回答四个基础问题。文件名的预算对象是 regular file 的 stem，也就是去掉最后扩展名后的 basename，目录前缀不参与 40 字符预算。扫描范围固定为 `src/`、`include/`、`tests/`，因为这三处承载生产实现、公共头文件和可执行测试。公共标识符只从 `include/` 头文件提取，避免把实现细节中的局部变量误称为公开合同。最后，baseline 不是“最多 740 个”的一行数字，而是 740 条具体文件路径和 883 个具体标识符组成的精确集合。

精确集合很关键。假设只检查 `count <= 740`，开发者可以删除一个旧长名，同时引入一个新长名，数量仍为 740，门就会误判为没有退化。当前实现会同时计算 `current - baseline` 和 `baseline - current`：前者是新增违规，后者是已经消失但尚未从基线清掉的陈旧项。任何一边非空都会失败。这样，替换式漂移不能躲在相同总数后面，真实修复又必须在同一提交更新 baseline，评审能直接看到删除了哪些债务。

## 三、入口、文件职责与数据结构

机械入口是 CTest 的 `elegance_name_census_contract`。CMake 使用 `minikv_add_standalone_source_dir_test` 构建 `minikv_elegance_name_census`，它只链接 C++ 标准库，不链接 `minikv` 运行时库。这个选择形成了很清楚的所有权边界：扫描器可以读取源码树，却不能通过链接 CommandProcessor、Store 或 TcpServer 意外执行项目逻辑。`MINIKV_SOURCE_DIR` 告诉它仓库根目录，`MINIKV_NAME_FILE_MAX=740` 与 `MINIKV_NAME_ID_MAX=883` 则是编译期 ratchet。

`tests/elegance_name_census_tests.cpp` 内部只有一个核心数据结构 `Census`，包含 `files` 和 `ids` 两个有序集合。选择 `std::set` 不是为了炫技，而是同时得到三项需要的性质：自动去重、稳定字典序和可以直接使用 `std::set_difference` 计算新增/消失集合。公共头文件里同一个长类型可能出现十次，但命名债务的对象是那个标识符本身，不应该因引用次数增加而把债务数夸大十倍。文件条目则保存相对仓库路径，所以两个目录中同名的文件仍是两个独立债务项。

`config/elegance-name-baseline.txt` 是纯数据合同。`F|` 表示文件路径，`I|` 表示公共头文件标识符；注释头记录预算和两类初始数量。它不包含自动 waiver，不支持通配符，也没有“忽略某个新目录”的开关。如此设计后，评审无需理解二进制格式或运行数据库，只需看一个普通 diff 就能知道基线是否收紧。生成模式 `--write-baseline` 仅用于按当前扫描结果机械重写文件，正常 CTest 不调用它；是否接受基线变化仍由源码 diff、硬上限和版本证据共同决定。

`docs/elegance-name-census.md` 是面向维护者的一页版本 note。它记录 40 字符预算、740/883 起点、扫描定义、需求到证据矩阵和复现命令。治理计划中的设计说明在实现之前写入，先声明抽象名、数据边界、行为边界和非目标，这对应新优雅门的 family 设计要求。`AGENTS.md` 中由评审者加入的 Elegance Gates 也随本版纳入版本控制，从而让未来会话不再依赖聊天上下文记忆这些规则。

## 四、公共头文件词法扫描如何避免虚假结果

直接对头文件执行 `[A-Za-z_][A-Za-z0-9_]*` 会产生大量假阳性。例如注释里对旧名称的解释、字符串里的 JSON 字段、字符字面量、原始字符串测试数据，以及 `#include` 路径中的长文件名都不是本次要统计的公共标识符。扫描器因此先经过两个明确阶段，再运行 identifier 正则。

第一阶段 `without_includes` 逐行跳过前导空白后的 `#include` 行。头文件路径本身已经由 filename census 统计，如果再次把路径 stem 当标识符，会造成同一债务重复计数。第二阶段 `strip_non_code` 使用小型状态机识别普通代码、行注释、块注释、双引号字符串和字符字面量，并处理反斜杠转义。对 C++ 原始字符串，它读取最多 16 字符 delimiter，找到对应的 `)delimiter"` 后整体跳过。清理结果保留代码字符并把非代码区域置为空格，随后正则只会看到真正的词法标识符。

这并不是完整 C++ 语义分析器，也不会声称知道一个类成员是 public、protected 还是 private。这里“公共头文件可见”采用的是更保守且可复现的定义：只要名字出现在 `include/` 的可编译代码区域，它就是外部阅读者和编译器可以观察到的头文件表面。这个定义会把私有成员也纳入存量，但不会漏掉模板参数、类型、函数和宏。比起实现一个容易错的半吊子 C++ parser，这个口径更适合作为仓库级 ratchet；真正改公共 API 时仍需单独做 ABI/源码兼容判断，本 census 不替代该评审。

## 五、通过、失败与收紧的精确机理

正常执行时，扫描器先生成当前 `Census`，再读 baseline。`compare_group` 对每一类数据做三层检查。第一层比较 baseline 数量和编译期 maximum，阻止有人在文本基线里直接加入更多豁免。第二层比较 current 数量和 maximum，形成最直观的只减不增上限。第三层比较两个精确集合，新增项会以 `+ path/name` 输出，陈旧项会以 `- path/name` 输出。只有数量没有增加且集合完全一致，CTest 才返回 0。

v1659 没有只验证绿色路径。建立基线后，验证过程临时复制出一个 stem 超过 40 字符的探针文件，使当前文件数从 740 变为 741。CTest 如预期返回失败，并明确打印 `file names census 741 exceeds ratchet 740` 和该探针路径。随后探针在同一验证命令的 `finally` 中删除，再次执行合同恢复绿色。这个负向实验说明失败门确实接到了当前文件系统，而不是读取基线后无条件打印成功。

未来 E-M2 改掉五个内部实现文件时，第一次运行应当先失败并列出五个 stale baseline 路径。这种失败不是故障，而是提醒执行者必须在同一提交运行受控的 baseline 更新，把文件上限从 740 收紧到 735。若改名后的新文件仍超过 40 字符，它们会同时出现在新增集合中，不能借删除旧项抵消。公共标识符没有变化时，其 883 上限保持不动；不能为了让版本数字“更漂亮”去改外部签名。

## 六、测试层次分别保护什么

第一层是扫描器自身的聚焦 CTest。它保护范围、预算、精确 baseline 和编译期 maximum，并通过故意注入的第 741 个文件证明红灯路径。第二层是 `project_docs_honesty_contract`，它要求 README、START_HERE 和能力快照指向 v1659 的 351 条测试与有界维护状态，防止文档继续停留在 v1658 的当前树描述，同时保留 v1658 外部 PASS 的历史事实。

第三层是完整 CTest。命名门本身不链接运行时，并不意味着版本只需跑一个测试。全量套件仍要覆盖 Store、WAL、Snapshot、RESP、TCP、OSFS、receipt parity、release fixture、coverage/workflow contract 和已有 E1-E10 门。它证明新增扫描器与 CMake 注册没有让其他目标漏编、重复命名或改变测试发现。第四层是 fixture 字节摘要对照：在 E-M2 改名之前冻结 `fixtures/release/*.json` 的聚合 SHA-256，改名后必须完全相同，避免内部文件移动意外伴随生成物重写。

真实 CLI smoke 仍有独立意义。即使 E-M1 没有生产代码变更，最终版本证据仍执行了只读 JSON 命令，确认 `read_only` 保持 true、`execution_allowed` 保持 false，SHARDJSON 仍没有 active router 或 write routing。测试数量和治理文档不能代替运行表面；反过来，一次 CLI 成功也不能证明命名 ratchet 有效，所以两类证据都要保留。

## 七、为什么 740/883 很高，却仍然是正确起点

这两个数字并不值得庆祝，它们说明历史治理对象命名确实过度携带上下文。但基线的职责不是美化成绩，而是如实冻结现实。如果为了让初始数字好看而排除某个大目录、只扫描最近文件，或把长公共符号一律称为 waiver，后续每个“减少五个”的百分比都会失去意义。740/883 是债务合同的起点，不是验收终点，也不是对这些名称合理性的认可。

本计划最多四版，也不会承诺在一个小批次里消除全部存量。原因有两层。技术上，很多长头文件和标识符已经是公共源码兼容表面，批量重命名可能让消费者无法编译；治理上，有限 top-5 能验证新门、改名流程、fixture parity 和 CI 是否形成可靠闭环，再由外部评审决定是否继续。硬停比无限期“再优化一点”更诚实，也符合当前 Java 收尾期间只允许有界维护的跨项目安排。

## 八、对下一版 top-5 的约束

当前排序最前的条目集中在 `shard_route_preview...compared_package_evidence_intake_audit` 家族。公共头文件路径和 namespace 名虽长，却属于当前公共表面，E-M2 不直接动它们。可安全选择的是只被 CMake 源列表引用的五个 `.cpp` 实现路径：主 audit、stages、slots、guards 和 validation。它们可以共享一个简洁的内部文件名前缀，仍然包含原公共头文件并实现原 namespace，因而链接符号、头文件路径、函数签名和输出字节都不变。

改名之前还要做两项核查。第一，用全仓库精确搜索确认旧 `.cpp` 路径没有被 fixture、归档合同、外部脚本或 waiver 点名。第二，在计划中先写不超过十行的 family 设计说明，明确“内部 package intake audit 实现文件”是抽象，公共 API 与证据字段是数据边界。完成改名后 baseline 文件计数必须 740 降到 735，identifier 仍为 883；如果结果不同，先解释 census，而不是调整规则迎合结果。

## 九、一句话价值总结

v1659 给 Node、Java 与 mini-kv 的松耦合协作增加的不是新权限，而是一条可复现、会失败、只会收紧的维护合同：它允许 C++ 内部逐步变短变清楚，同时把 CLI、fixture、公共 API、no router、no write 与 no execution 边界留在原位。
