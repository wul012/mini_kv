# v1633 OSFS 课程设计最终交付与质量门槛讲解

## 本版目标、角色和不是做什么

v1633 的目标不是再给 OSFS 文件系统加一个炫目的命令，也不是把前两版已经能跑通的内容包装成一个好看的文件夹就急着交付。本版的角色更接近一次交付审计：把 v1631 的持久化用户表、MFD/UFD 二级目录、登录与权限隔离，以及 v1632 的 fd 读写位置、range I/O、`SEEK/TELL` 和多模式打开行为，重新对照操作系统课程设计要求，冻结成一个可以复现、可以审阅、可以提交的最终材料包。它要回答的问题很朴素：老师或同学拿到这个目录后，能不能看懂系统做了什么，能不能沿着脚本跑出同样结果，能不能在报告里看到每个必做点背后的真实实现，而不是只看到命令清单。

所以本版明确“不是什么”。它不是新的 KV 存储主线，不把 OSFS 接进 WAL、snapshot、TCP、RESP 或生产 readiness 证据链；它不是跨项目 managed audit 的一个 receipt，也不读取 credential、不解析 raw endpoint、不启动 Node/Java、不承担 audit/order authority；它也不是通过截图或报告掩盖实现短板。v1630 曾经给过一个可运行草稿，但那时的目录模型仍然偏向单根目录，登录和用户表没有真正落盘，fd offset 也没有达到课程要求里的文件描述符语义。v1633 的核心价值，正是把这类“看起来像完成，细问又不够”的地方拉回实现、测试和材料三条线上重新闭环。

## 入口和材料结构

本版有两个入口。运行入口仍然是 `minikv_osfs`，由 v1631/v1632 的 C++ 实现支撑；材料入口是 `课程设计交付/v1633-osfs-final/`。后者不是随便堆文件，而是按提交、复现、审查三种读者路径组织。

`README.md` 面向第一次打开目录的人，先说明这是课程要求里的“实验二 Linux 二级文件系统”，并把 v1630 单根目录草稿降级为历史草稿。`osfs-demo-script.txt` 是真实命令脚本，`osfs-demo-transcript.txt` 是用当前可执行程序跑出来的完整输出，不靠手写模拟。`演示步骤.md` 给出复现实验的顺序，`提交清单.md` 告诉最终提交时应交哪些文件，`需求-证据矩阵.md` 则是本版质量门槛最关键的文件：每一条课程要求都要能找到对应实现、测试或演示证据，并且要写清楚哪些是教学简化，而不是把简化点藏起来。

报告生成入口是 `tools/build_osfs_coursework_report.py`。这个脚本承担两件事。第一，它生成三张图：系统结构图、磁盘布局图和 fd 描述符流程图。图不是为了装饰，而是把隐藏在二进制镜像里的结构翻译成人能扫读的模型。第二，它生成 `操作系统课程设计报告-OSFS-v1633.docx`，再由 LibreOffice 导出 PDF，并用 Poppler 渲染检查最终页面。课程设计报告最怕的是“内容在文件里，但页面上挤坏了、表格断裂、图片压扁、引用落到空白页”。v1633 的脚本和渲染检查就是为了解决这个最后一公里问题。

## 响应和证据结构

OSFS 运行时的响应结构是命令行文本，但 v1633 对它的要求比“能打印 OK”更严格。脚本先用 `WHOAMI` 和未登录的 `CREATE` 证明默认状态没有隐式用户身份，随后用错误密码登录证明认证失败会被拒绝。`LOGIN alice alice123` 后，Alice 创建 `report`，以 `rw` 模式打开并连续写入 `course-` 和 `design`，输出里的 write offset 从 7 到 13。接着通过 `SEEK 4 0`、`READ 4 6`、`READ 4 1` 和 `READ 4` 证明同一个 fd 会保留独立读位置，读取不是每次都从文件开头重来。

权限和二级目录的证据也在同一条 transcript 里完成。Alice 对 `report` 执行 `CHMOD 0600` 后退出，Bob 登录后看到自己的 UFD 为空，并且 `OPEN alice/report r` 返回拒绝。这一步很重要，因为它区分了真正的 MFD/UFD 与“根目录中过滤 owner”的伪实现。如果只是按当前用户过滤一个平面目录，重启、root 审查、跨用户路径和 UFD 行为很容易露出破绽。v1631 已经把用户表、MFD、每用户 UFD 放进磁盘格式 v2，v1633 的 transcript 则把这种结构用用户可见行为暴露出来。最后 root 登录后执行 `DIR alice`，再打开并读取 `alice/report` 得到 `course-design`，说明 root 审查路径不是临时拼接出来的文本，也不是 Bob 权限绕过。

材料证据分三层。第一层是源码和测试，主要在 `include/minikv/osfs/filesystem.hpp`、`src/osfs_disk_layout.*`、`src/osfs_directory.cpp`、`src/osfs_file_io.cpp`、`src/osfs_command_processor.cpp` 和 `tests/osfs_tests.cpp`。第二层是可运行证据，即 focused CTest、full CTest 和真实 CLI demo 输出，对应 `f/1633/图片/` 下的截图。第三层是交付证据，即最终 DOCX/PDF、需求矩阵、演示脚本、演示 transcript、提交清单和 README。三层缺一不可：只有代码没有报告，交付不完整；只有报告没有测试，可信度不够；只有截图没有可复现脚本，下一次维护时很难判断是否仍然成立。

## 核心流程如何闭环

本版的闭环从原始要求开始，而不是从已有代码开始。课程 PDF 要的是 Linux 二级文件系统模拟，关键教学点包括目录结构、用户登录、文件描述符、读写位置、权限和磁盘空间管理。v1633 首先承认 v1630 不够，把旧包标注为 superseded，再把最终入口迁移到 `v1633-osfs-final`。这一步看似只是文档文字，实际是在交付语义上纠偏：历史材料可以保留，但不能继续冒充最终提交物。

第二步是用真实程序生成 transcript。这里没有把命令输出重新誊写进报告，而是执行 `minikv_osfs --disk ... --format --blocks 96 --script ...` 得到完整输出。脚本覆盖登录前阻断、错密码、Alice 写入与 offset、权限修改、Bob 隔离、root 审查。它的输出被放进最终目录，也被报告引用。这样做的好处是报告中的“系统支持某行为”有对应命令和返回值，读者不需要相信作者的描述。

第三步是重建报告和图。`tools/build_osfs_coursework_report.py` 不再沿用 v1630 的单根目录叙述，而是把磁盘布局改成 superblock、block bitmap、inode bitmap、user table、inode table、MFD、UFDs 和 data blocks 的 v2 模型。fd 流程图也从“打开后读写文件”升级为“open 产生 fd，fd 保存读写位置，read/write/seek/tell 修改或读取位置，close 释放 descriptor”。这些图把实现机制摊开，避免报告只有文字口号。

第四步是渲染验收。DOCX 生成后导出 PDF，PDF 再渲染成页面 PNG 和 contact sheet。v1633 最终检查的是 9 页报告，重点看图表是否清晰、分页是否正常、需求矩阵是否被切坏、参考文献是否被挤到空白页。文档任务不能只看文件是否存在，因为 `.docx` 能打开不代表适合提交；PDF 能导出也不代表页面可读。本版把这件事纳入归档截图，是为了以后如果有人质疑报告质量，可以直接看到渲染结果。

## 边界字段和项目边界如何理解

OSFS 课设不属于 mini-kv 原有生产证据链，因此这里的边界字段不是 JSON receipt 里的固定字段，而是交付矩阵和 README 里的显式约束。`read_only` 在本版语境中表示跨项目层面只读：v1633 不消费 Node/Java 的运行服务，不打开 managed audit connection，不写 approval ledger。`no_network` 表示课程设计 demo 只跑本地二进制和本地磁盘镜像，没有 HTTP/TCP 请求，也没有启动 `minikv_server`。`no_write` 不能误解为 OSFS 内部不写文件，因为文件系统实验当然要写自己的教学磁盘镜像；它真正约束的是不写原 KV store、不写 WAL、不写 snapshot、不写跨项目 audit/order 状态。`no_execution` 也不是禁止本地 OSFS demo 执行，而是禁止把这个版本当成 LOAD、COMPACT、RESTORE、SETNXEX 或跨项目治理命令的执行入口。

这些边界说明很必要。mini-kv 项目长期有大量 readiness、receipt、non-participation、no-start、no-write 证据，OSFS 课程设计则是一个相对独立的教学层。如果不写清楚，后续维护者可能会误以为课程设计功能可以直接进入生产 KV 命令主线，或者把课设报告里的“文件系统写入”误读成 WAL/restore 执行能力。v1633 的写法是把教学写入限定在 `minikv_osfs` 的磁盘镜像内，保持与 Store、WAL、snapshot、TCP/RESP 完全隔离。

## 测试保护了什么

focused 测试主要保护 OSFS 自身的关键行为。v1631 的测试证明用户表持久化、登录失败、目录隔离、root 审查、权限拒绝和重新打开后的二级目录状态都成立。v1632 的测试进一步证明 fd offset 不是命令层假象：顺序读会推进读位置，连续写会推进写位置，`rw` 可以在原位置覆盖，`a` 会追加，EOF 不会越界，跨 512 字节块的 range I/O 能读写正确内容，重开后内容仍在。它们共同回应了课程设计里最容易被老师追问的几个点：目录是不是二级的，用户是不是落盘的，权限是不是有效的，fd 是否真有独立位置，文件内容是否跨块和重启后仍一致。

full CTest 的意义是保护隔离边界。OSFS 是新加的教学层，但 mini-kv 还有原 KV 命令、WAL、snapshot、RESP、fixture、release evidence 和大量治理测试。全量 344/344 通过说明 OSFS 的新增文件和报告生成没有破坏原项目行为。特别是本版最后只改材料和生成器，仍然保留 full CTest 证据，是为了避免“文档收口”悄悄污染代码或 fixture。对这个项目来说，局部正确不够，必须同时证明主线没被带偏。

报告渲染检查保护的是另一类风险。课程设计交付不是只给机器看的，最终有人会打开 Word/PDF 阅读。v1633 通过 LibreOffice 导出 PDF，再用 Poppler 渲染页面截图，确认 9 页报告没有空白尾页、没有表格明显断裂、没有图片遮挡文字、没有引用丢失。这个步骤不属于 C++ 单元测试，却是课程交付质量的一部分。用户前面指出“不要盲目给低劣品交付”，本版把这个要求落实到了流程里。

## 设计取舍和仍然披露的限制

v1633 没有把 OSFS 做成完整类 Unix 文件系统。它保留单组权限模型，没有实现真实 group database；采用直接块，不做间接块；没有 fsck、崩溃恢复日志、journaling 和并发锁；密码哈希是教学级实现，不宣称生产安全。这些限制不是临时忘写，而是被放进需求矩阵和报告边界里。课程设计的评分重点是二级目录、登录、fd、权限、空间分配和可演示性，把系统扩成生产文件系统反而会偏题，也会把 mini-kv 主线拖进不必要的复杂度。

更重要的是，本版没有用版本纪律压低交付标准。v1631 负责补齐目录和用户，v1632 负责补齐 fd offset，v1633 才进入最终材料。这样拆分比在一个版本里塞满所有变化更稳：每一版都有自己的测试和归档，回看时能知道哪个行为从哪一版开始成立。v1630 被保留但标注 superseded，也体现了一个维护原则：历史证据路径稳定，但错误的最终口径要及时降级。

## 本版带来的维护价值

对后续维护者来说，v1633 最大的价值不是多了一个 PDF，而是建立了一种“交付前反向审查”的模板。以后再做课程设计、演示包或跨项目材料，不应该先写漂亮报告再想证据，而应先列要求矩阵，再用实现、测试、真实运行和渲染检查逐条填实。OSFS 在 mini-kv 中仍是独立教学层，Node/Java/mini-kv 的松耦合融合也因此更清楚：mini-kv 可以提供本地、可复现、边界明确的学习型能力和证据材料，但不会因为一个课程功能就变成上游订单系统、审计系统或外部网络执行器。
