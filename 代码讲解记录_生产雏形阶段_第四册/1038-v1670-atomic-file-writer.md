# v1670 代码讲解：把 WAL 与 Snapshot 的文件提交协议收成一个可靠状态机

## 一、本版到底解决什么问题

v1670 的目标不是给 mini-kv 增加一条新命令，也不是改变 WAL2 或 Snapshot 的磁盘格式，而是偿还持久化边界里一类很容易被忽视的重复。重构前，`wal.cpp` 和 `snapshot.cpp` 各自维护一套临时文件保护类、临时路径生成函数、Windows/POSIX 文件替换函数，以及几乎相同的 `flush -> close -> replace` 控制流。两份代码都能工作，但它们表达的是同一个基础协议：先把完整内容写到目标文件所在目录的临时文件，只有写入和关闭都成功后，才用一次平台替换操作把临时文件切换为正式目标。

这种重复的危险不在于多几十行代码，而在于协议会产生两个事实来源。将来若只给 WAL 修复临时文件碰撞，只给 Snapshot 增加父目录校验，或者只在一边调整 Windows 替换选项，两条持久化路径就会悄悄分叉。分叉通常不会在正常写入时暴露，而会在磁盘错误、路径冲突、进程异常或并发保存时暴露。v1670 因而提取了私有的 `minikv::detail::AtomicFileWriter`，让它成为单个目标文件提交生命周期的唯一所有者。

本版明确不是什么：它不是新的公开 API，不增加命令路由，不接管 WAL 记录编码或 Snapshot 键值编码，不改变自动 compact 决策，不触碰 OSFS 镜像写入，也不赋予 Node 或 Java 任何执行权限。这里的“atomic”只描述同目录临时文件完成后对一个目标路径的替换可见性，不表示跨多个文件的事务，更不表示数据库已经获得完整的掉电持久性保证。

## 二、重构前的真实重复在哪里

重构前的 WAL 所有者拥有 `TempWalFile`，Snapshot 所有者拥有 `TempSnapshotFile`。二者都在析构时尝试删除未提交的临时文件，都通过“正式文件名 + 特定 tag + 时间信息”构造临时路径，也都分别实现 Windows `MoveFileExW` 与 POSIX rename。保存流程则都是：创建父目录、打开临时输出流、写领域内容、刷新、关闭、替换，任何一步失败就返回 false。

表面上，WAL compact 写的是 `WAL2` 头、操作数量与编码后的操作；Snapshot save 写的是 `SNAPSHOT` 头、键数量、键值和过期时间。两者的数据结构不同，所以编码必须继续留在各自 owner 中。但是从“编码完成”开始，两者面对的是完全相同的资源状态：一个尚未提交的临时文件、一个不能提前破坏的旧目标，以及一次只能成功一次的提交动作。v1670 提取的是这一段稳定机制，而不是把不同领域概念塞进万能模板。

重构后，`wal.cpp` 和 `snapshot.cpp` 不再出现旧临时保护类、临时路径 factory、`MoveFileExW` 或本地 replace helper。它们只负责把正确的字节写入 `writer.stream()`，然后调用 `writer.commit()`。测试中的 source-ownership contract 会直接读取两个 owner 源文件：若旧类名、旧路径函数或平台替换实现回流，测试立刻失败。这使“唯一所有者”不再只是评审意见，而是可执行的架构约束。

## 三、`AtomicFileWriter` 的职责边界

新类位于 `src/atomic_file_writer.hpp` 与 `src/atomic_file_writer.cpp`，命名空间是私有语义明显的 `minikv::detail`。头文件没有进入 `include/minikv/`，因此不会扩张公共库契约。类只持有四类状态：正式目标路径、临时路径、一个 `std::ofstream` 和 `committed_` 标记。它不知道自己承载的是 WAL、Snapshot 还是别的文本内容，也没有 `kind` 分支、回调注册表或领域枚举。

构造阶段先检查目标的父路径。父目录不存在时，writer 会用 `create_directories` 创建；父路径被普通文件阻塞、创建报错或最终不是目录时，writer 保持未就绪，不会再伪装成可写状态。随后它在同一父目录生成临时路径并打开输出流。调用者必须先检查 `ready()`，只有流已打开且状态良好时才开始编码。

同目录不是装饰性选择。多数平台的 rename/replace 原子语义要求源与目标位于同一文件系统；若把临时文件放进系统临时目录，跨卷移动可能退化为复制加删除，提交边界就失去原来的意义。临时名由目标文件名、owner 传入的稳定 tag、进程号、单调时钟计数和进程内原子递增序号组成。Snapshot 保留 `.tmp.` 标识，WAL compact 保留 `.compact.tmp.` 标识，因此运维排查和既有测试仍能辨认临时文件类型；进程号加原子序号则降低同一时刻多次写入的碰撞风险。

`temp_path()` 只用于私有基础设施测试观察生命周期，不把路径暴露到产品公共接口。复制和移动都被删除，是因为这个对象代表一份独占资源与一次性提交能力；若允许隐式复制，两个析构函数可能争抢同一临时路径，若允许随意移动，又会让 moved-from 状态成为额外维护面。这里用较小的类型表面换取更清楚的资源所有权。

## 四、提交状态机如何运行

可以把这个类理解成三个业务可见状态。第一是“未就绪”：父目录无效或临时流无法打开，此时 `ready()` 为 false，`commit()` 必须失败。第二是“已准备但未提交”：临时流可写，调用者把完整领域内容编码进去；若作用域在这一阶段结束，析构函数关闭流并尽力删除临时文件。第三是“已提交”：flush、close 和平台 replace 都成功，`committed_` 被置为 true，析构时不再删除已经成为正式目标的路径。

`commit()` 先拒绝重复提交和未打开流。随后按顺序执行 flush、检查流状态、close、再次检查流状态，最后调用平台替换。任何一步失败都返回 false，且不会把 `committed_` 置真。对象离开作用域时，析构函数便清理仍存在的临时文件。只有 replace 成功后才设置提交标记，所以“返回 true”与“正式路径已经切换”保持同一个事实。

重复调用 `commit()` 返回 false，而不是把第一次成功当作幂等成功。这是刻意的状态约束：调用方若意外提交两次，程序应该尽早看见生命周期误用，而不是吞掉错误。直接测试先写入旧目标，再通过 writer 写入新内容并提交，断言目标内容确实被替换、临时路径消失、第二次提交失败。这样同时保护了数据结果、资源结果和状态机结果。

## 五、失败路径为什么比成功路径更重要

只测“保存成功”无法证明原子写入抽象有价值。本版把失败面分成 writer 直属故障和 owner 集成故障两层。直属测试覆盖主动放弃、坏流、父目录自动创建、父路径被文件阻塞、目标本身是目录导致 replace 失败，以及连续 writer 的临时路径唯一性。

主动放弃测试创建 writer 并写入内容，却不调用 commit。作用域结束后断言旧目标未变且临时文件不存在，证明 RAII 清理不是注释。坏流测试人为设置输出流失败状态，再调用 commit，断言提交失败、旧目标不变、临时文件最终被清理。父目录测试一边证明合法的多级父路径可自动建立，一边证明普通文件阻塞父路径时 writer 不会就绪。目录目标测试让临时文件写入成功但最终 replace 必然失败，专门验证最接近提交点的红路仍不会泄漏临时文件。

owner 层还分别给 Snapshot 与 WAL 增加了失败哨兵。测试先在目标目录中放置不会被误认成有效数据的计数哨兵，再把目标路径设置成目录，使真正的保存或 compact 在最终替换处失败。断言不仅要求 API 返回 false，还要求原有哨兵值保持不变、目标目录继续存在、同类临时文件数量为零。这样可以防止一种常见的假修复：为了兼容 Windows，先删除旧目标再 rename。那种做法在 rename 失败时会丢掉旧数据，本版明确把它列为失败条件。

WAL 成功 compact 测试也检查 `.compact.tmp.` 没有残留，Snapshot 原有保存与加载测试继续验证编码和恢复。直接 writer 测试回答“基础设施的状态机是否正确”，owner 测试回答“真实调用点是否正确接线”，既有持久化套件回答“领域语义是否保持”。三层证据彼此不能替代。

## 六、Windows 与 POSIX 的替换语义

Windows 分支继续使用 `MoveFileExW`，标志为 `MOVEFILE_REPLACE_EXISTING | MOVEFILE_WRITE_THROUGH`。前者允许用完成的临时文件替换已有目标，后者要求系统在函数返回前尽力完成移动操作相关的刷新。路径使用宽字符接口，避免退回窄字符 Windows API。POSIX 分支通过 `std::filesystem::rename` 的 `error_code` 重载完成同目录切换，失败以 false 返回，不用异常跨越 owner 边界。

两边共享的是“成功才切换目标”的控制协议，不是假装底层文件系统完全一致。特别需要诚实说明：当前 writer 对输出流执行 flush 和 close，但没有直接取得文件描述符执行 POSIX `fsync`，也没有在 rename 后同步父目录；`MoveFileExW` 的 write-through 也不能把所有硬件、缓存和断电模型压成绝对保证。因此，原子替换不等于完整掉电耐久。它显著改善的是单目标文件的可见性与失败清理，不能被描述为数据库事务日志已经在任意断电点绝不丢失。

同理，它不协调 WAL 与 Snapshot 两个目标的共同提交。若未来需要跨文件一致性，必须另行设计带版本、意图记录或恢复协议的事务层，而不是给这个类增加一个 targets 数组。保持抽象只解决一个清晰问题，是本版优雅度提升的重要部分。

## 七、WAL 与 Snapshot 仍然各自负责什么

`Wal::compact()` 仍然决定哪些操作应该写回，仍然输出 WAL2 格式，仍然维护 compact 成功后的内部计数。它只把原来的本地临时流换成 `AtomicFileWriter` 的流，并传入 `compact` tag。只有 `commit()` 返回 true 后，compact 才能更新成功状态。writer 完全不知道 WAL 操作、转义规则、自动 compact 阈值或 replay 语义。

`Snapshot::save()` 仍然按原顺序遍历 store snapshot，过滤已经过期的键，编码键值和可选过期时间，并返回保存条目数。它使用默认 `.tmp.` tag，但没有把 Snapshot 特有计数塞进 writer。`Snapshot::load()` 没有迁移，因为加载是读取与解析路径，不属于临时文件提交协议。这样的分界使两个 owner 变短，却没有把领域逻辑挪进一个难以命名的“持久化万能工具”。

本版源代码整体是净减少：两个 owner 删除了重复类、路径 factory、平台 include 和提交控制流，新 writer 只保留一次实现。新抽象不是为了让类图更漂亮，而是为了把一个具有失败语义的协议变成唯一事实来源。文件体积、命名长度和功能版本新增源码预算也都受既有优雅门约束。

## 八、测试与机械门分别保护什么

新注册的第 354 个 CTest 名为 `atomic_file_tests`。它集中保护 writer 的成功、重复提交、放弃、坏流、父目录、替换失败、唯一路径和 owner 源码归属。源码归属检查并不验证运行结果，却能阻止未来维护者把 `MoveFileExW` 或第二份 Temp 类悄悄复制回 WAL/Snapshot。运行测试与结构测试组合起来，才能同时保护行为和设计。

`wal_tests` 新增失败替换和临时文件零残留断言，`snapshot_tests` 新增对应的保存失败哨兵。原有 WAL replay、compact、Snapshot save/load 和过期语义测试全部继续执行。更宽的恢复集合还包含 command persistence、recovery evidence、restore、TTL 和 index 测试，防止“局部类测试全绿、系统恢复路径却漂移”。在最终归档前，本版先完成了全目标构建，耗时约 629 秒；随后 15 个持久化与恢复重点测试全部通过。这个较长构建时间也成为下一版评估测试编译拓扑的真实基线，而不是凭感觉宣称构建慢。

CI 的 gcovr core filter 已加入 `atomic_file_writer`，对应 contract expectation 同步更新。这样抽出的代码不会因为换了文件就退出覆盖率统计。README 对覆盖率的表述也保持校准：2345/2122/90% 是 v1658 经评审的历史基线，当前 filter 已扩展，真正的新总数由远端 coverage job 重新计算；在 CI 返回前不伪造本地 Windows 无法产生的 Linux gcovr 数字，90% floor 则继续作为会失败的门。

## 九、真实命令链应当证明什么

最终 smoke 不只运行一个 SAVE。它会在真实 `minikv_cli` 中写入两个键，保存 Snapshot，修改其中一个键，再 LOAD 回 Snapshot，验证旧值恢复；接着执行 COMPACT，关闭并以同一 WAL 重新打开，验证 replay 后两个键仍可读取。同时检查工作目录中不存在 `.tmp.` 或 `.compact.tmp.` 残留。这个链条分别触达 Snapshot 成功提交、Snapshot 读取、WAL compact 成功提交和进程重开恢复。

这里仍要区分证据强度。CLI smoke 证明真实二进制和真实文件系统上的主路径可运行；直属故障测试证明难以通过 CLI 稳定制造的红路；全量 CTest 证明重构没有伤害旁路契约；CI 再证明 Linux、macOS、Windows、sanitizer、coverage、format 和 archive 门。截图只是把这些命令的关键结果归档，不能取代命令本身，更不能靠一张绿色图片声称未执行的边界已经通过。

## 十、与跨项目边界的关系

WAL 与 Snapshot 是 mini-kv 自己的存储恢复机制。Node 可以读取冻结 fixture 或通过受控 CLI 获取只读证据，Java 仍是订单域权威；本版没有增加网络路由、写路由、恢复执行入口、生产 secret 或审批绕过。`read_only`、`execution_allowed`、`order_authoritative`、`mutates_store`、`touches_wal` 等控制面字段没有任何变化，receipt fixture 也不因内部文件提交重构而重写。

对松耦合集成而言，本版的价值是让 mini-kv 自己生成持久化证据时少一个实现分叉点，同时保持外部消费者看到的命令、格式、摘要和权限边界完全稳定。它提高的是内部可靠性和可维护性，不借机扩大系统权力。

## 十一、维护者下一次改动时怎样判断放在哪里

如果改动涉及 WAL2 操作编码、compact 选择或 replay，应该留在 WAL owner；如果涉及 Snapshot 键值编码、过期时间或 load，应留在 Snapshot owner；如果问题是“完整内容如何从临时路径安全切换为单个正式目标”，才进入 `AtomicFileWriter`。若出现第三个真正相同的单文件提交 owner，应复用此类并增加 owner 集成测试，而不是复制 helper；若第三个场景需要跨文件事务、随机访问二进制写入或目录级交换，则说明抽象边界不同，不能为了满足三次规则硬塞进去。

失败处理同样有判断标准：writer 负责不提前破坏旧目标、失败后清理自己的 temp；owner 负责只有提交成功才更新领域计数或返回成功；更高层负责决定是否重试、报告错误或进入恢复流程。三层各自拥有可解释的状态，就不会出现一个工具类同时决定业务、日志和文件系统政策。

## 十二、一句话总结

v1670 用一个小而私有、故障路径可验证的提交状态机替代 WAL 与 Snapshot 的双份实现，使 mini-kv 的单文件持久化切换更一致，同时不改变任何跨项目合同、公开格式或执行边界。
