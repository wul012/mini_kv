# v1670 命令证据说明

本目录只保留四张互相补充的摘要图。完整 720 行 CTest 输出、两个真实 smoke 工作目录、临时证据页面和本地构建树仅用于验证，版本收尾时删除；归档不保存二进制或原始调试日志。

## 01-atomic-architecture.png

对应源码所有权与体积复算。WAL/Snapshot 中旧的两份 Temp class、两份 path factory、本地 replace helper 和 `MoveFileExW` 引用全部归零，两个 owner 各保留一次 `AtomicFileWriter` 使用。新私有 writer 共 119 个非空行，连同 owner 删除后，相关产品源码从 642 降到 612 个非空行，净减少 30 行。该图证明本版是把失败协议收成唯一事实来源，而不是增加一层更重的包装。

## 02-failure-matrix.png

对应 `atomic_file_tests`、`wal_tests`、`snapshot_tests` 以及文档/架构/恢复聚焦集合。直属测试覆盖成功替换、重复提交、放弃、坏流、父目录、目录目标替换失败和唯一路径；owner 测试再验证失败时旧哨兵不变且临时文件为零。聚焦结果 17/17，通过的 source-ownership ratchet 会拒绝重复 helper 回流。

## 03-full-verification.png

对应本版全目标构建、最终全量 CTest、格式门与文档门。干净全目标构建完成 1056 个步骤，约 628.9 秒；在中文讲解和当前文档稳定后，全量结果为 354/354、0 失败、180.07 秒。`fixtures/release` 和 `include/minikv` diff 都为空，说明没有靠改历史证据或公共 API 让重构通过。

第一次全量命令的外层等待时间设置过短，外壳超时而子进程继续，随后显式长超时的完整运行得到上述绿色结果。这是本地命令编排重叠，不是产品或测试失败。

## 04-real-cli-reopen.png

真实二进制依次执行两个 SET、SAVE、修改、GET、LOAD、GET、COMPACT、WALINFO 和 QUIT，再使用同一 WAL 重开。结果证明 Snapshot 把 `alpha` 从 changed 恢复为 one，compact 写回两条存活记录，重开重放 2 条且读取到 `alpha=one`、`beta=two`，skipped/truncated/checksum-failed 均为 0，目录内没有 `.tmp.` 或 `.compact.tmp.` 残留。

第一次 smoke 检查器错误地要求值从行首出现，忽略了真实 CLI 的 `mini-kv>` 提示符，因此主动报红。随后在全新目录中使用提示符感知断言重跑并通过，产品代码没有为检查器让步。

## 边界结论

四张图分别证明唯一实现、红路安全、全仓库无回归和真实进程重开恢复。这里的 atomic 只表示同目录单目标替换，不等于跨文件事务，也不宣称已经具备 POSIX `fsync` 加父目录同步意义上的完整掉电耐久；外部命令、fixture、router、network、write、restore execution 和 order authority 均未改变。
