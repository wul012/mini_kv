# v1635 命令归档说明

v1635 的主题是把 OSFS 课设层从“能登录、能读写、能跨直接块容量”继续加厚到“能自检、能管理真实落盘用户”。本目录只归档 v1635 的构建、测试和真实命令证据；它不代表 mini-kv 主线 KV、WAL、snapshot、TCP/RESP、Node/Java 证据链发生变化，也没有引入执行型修复器、自动恢复、二级/三级间接块、崩溃日志或跨项目路由。

## 01-configure-build.png

命令：`cmake -S . -B cmake-build-v1635 ...`，随后执行 `cmake --build cmake-build-v1635`。

这张图证明 v1635 在独立构建目录中重新配置和编译。源码新增了 `src/osfs_fsck.cpp` 与 `src/osfs_user_admin.cpp`，并把它们接入 `CMakeLists.txt`；如果不重配和重编，后续测试可能误用旧的 `minikv_osfs` 或旧的 `osfs_tests`。构建输出里的 Windows 长路径 warning 是本仓库历史长测试目标名导致的本地提示，不是 v1635 新增源文件失败；最终 `ninja: no work to do` 表明截图前构建产物与源码一致。

## 02-focused-osfs-tests.png

命令：`ctest --test-dir cmake-build-v1635 -R '^osfs_tests$|^osfs_cli_smoke$' --output-on-failure`。

这张图证明 OSFS 聚焦测试 2/2 通过。`osfs_tests` 覆盖新增的 clean FSCK、坏 block bitmap 检出、root-only `USERADD`、root/self `PASSWD`、旧密码失效、新密码重开可登录、Bob 越权改密/看目录失败、root 可审查 Carol UFD、用户表满时失败以及最终一致性仍 OK。`osfs_cli_smoke` 继续保护原有课程 shell 的登录、目录、文件读写路径没有被新命令打坏。

## 03-full-ctest.png

命令：`ctest --test-dir cmake-build-v1635 --output-on-failure -j 4`。在归档前还跑过一次串行全量 CTest，结果同样是 344/344 通过。

这张图证明 v1635 没有把 OSFS 课设层扩展外溢到 mini-kv 主线。全量测试覆盖 store、command、shard readiness、runtime receipts、WAL、snapshot、RESP/TCP、archive inventory、code walkthrough gate、no-network/no-write/no-execution 证据链等 344 个测试。由于 DevTools MCP 的全页截图在长 `pre` 页面上超时，且首次视口截图只拍到开头没有拍到最终摘要，本图使用同一份 UTF-8 CTest 日志进行本地文本渲染兜底，把关键测试节点和 `100% tests passed, 0 tests failed out of 344` 放在同一张图里。

## 04-useradmin-fsck-smoke.png

命令：`minikv_osfs --disk useradmin-fsck-smoke.img --format --blocks 192 --script 04-useradmin-fsck-smoke-script.txt`。

这张图来自真实 `minikv_osfs` 会话的 UTF-8 transcript，而不是测试 mock。流程先由 root 登录并执行 `FSCK`，证明干净镜像返回 `FSCK OK checks=6`；随后 root 执行 `USERADD carol carol123` 和 `PASSWD carol carol456`，Carol 的旧密码被拒绝、新密码可登录；Carol 创建 `diary` 并写入 11 字节后再自助改密，旧密码再次失败、新密码成功；Bob 登录后尝试 `PASSWD carol blocked` 和 `DIR carol` 都被拒绝；最后 root 能列出 Carol 的 UFD，并且最终 `FSCK` 仍为 OK。由于 MCP 视口截图无法把完整长会话拍到同一屏，本图按真实 transcript 提取验收断言渲染，原始 transcript 仅作为临时证据参与生成，不长期保留。

## 05-fsck-corruption-smoke.png

命令：先用 `minikv_osfs` 创建 `damaged` 文件并通过 `STAT` 读出物理块，再直接翻转镜像中该物理块的 block bitmap 位，最后在同一镜像上运行 `FSCK`。

这张图证明 `FSCK` 不是只会打印 OK 的装饰命令。坏盘面演示把文件数据块 20 的 bitmap 位从 used 改为 free，随后 `FSCK` 输出 `FSCK ERROR checks=6` 和 `ERROR block bitmap: block bitmap mismatch at block 20`。其它检查项仍为 OK，说明检查器能定位局部不一致，而不是遇到任意异常就泛化报错。这里仍然是只读检查：它报告错误，不自动修复、不改位图、不移动文件、不释放块。
