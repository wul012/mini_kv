# mini-kv v1652 命令归档说明

本版把 v117-v120 四个 legacy resolver receipt formatter 迁移到 ordered JSON builder，并把机械 census 提升到 Claude review floor。四张截图都来自代码、讲解和维护文档稳定后的最终验证结果，由 Playwright MCP 生成并经过逐张视觉检查；没有把早期构建、逐文件试验或中途状态当作最终证据。

## 01-builder-migration

文件：`图片/v1652-01-builder-migration.png`

该图概括生产代码变化。旧实现由单个巨型 `return` 表达式同时维护字段名、转义、逗号、嵌套对象和无大括号 boundary fragment；新实现以 `OrderedJsonField` 保存有序顶层字段，并命名 source、archive、decision、plan、interface shape、fake wiring、summary 和 endpoint 对象。

历史 archive files、candidate decisions 与 checks 等完整 JSON value helper 仍保持原样。四个 receipt 的边界字段因阶段语义和顺序不同，由四个本地无参数 appender 所有，没有引入带大量布尔参数的万能 schema。图底部列出的 no credential、no endpoint、no network、no write、no execution、no authority 是本版不允许突破的硬边界。

## 02-parity-census

文件：`图片/v1652-02-parity-census.png`

该图对应 v1651 独立提交的 `runtime_receipt_legacy_resolver_chain_baseline_tests` 和 `receipt_builder_census_contract`。四份 runtime 对象继续与 frozen fixture 逐字节相等，字节数、顶层字段数和 v1651 digest 全部不变，双方 `\\u0027` 次数均为零，没有 canonical 或 whitespace waiver。

census 的可复现输出是 `sources=28 formatter_owners=27 builder_backed=18 pending=9 waivers=1`。CMake floor 从 14 收紧到 18；测试或文档增长不能冒充生产迁移，任何一个已迁移 owner 退回旧拼接都会使该门失败。该结果正式到达 Claude review checkpoint。

## 03-final-verification

文件：`图片/v1652-03-final-verification.png`

最终 all-target build 重新生成并链接 `minikv` 静态库、CLI/server、OSFS 与全部测试 executable。第一次全目标构建超过工具的五分钟调用窗口时已到约 91%，子构建进程继续正常收尾；待同一组 PID 退出后，立即重跑短增量全目标构建，得到 exit 0 和全部目标 100%。这是命令包装超时，不是编译失败，期间没有启动第二个并发 build。

最终 focused CTest 包含 census、通用 builder、v1651 oracle、四个领域 receipt 和 SMOKEJSON/runtime-smoke/manifest 三个下游表面，共 10/10 通过，耗时 2.43 秒。最终 full CTest 为 347/347，耗时 162.33 秒，覆盖 Store、WAL、snapshot、RESP、OSFS、历史 evidence 和文档契约。fixture 与测试期望均未修改。

## 04-cli-boundaries

文件：`图片/v1652-04-cli-boundaries.png`

真实 CLI 输入依次为 `SMOKEJSON`、`CHECKJSON LOAD data/prod.snap`、`QUIT`。四个 migrated receipt marker 在完整 SMOKEJSON 中各出现一次，说明对象仍从真实聚合路径暴露，不只是在 formatter 单测中存在。

`read_only=true`、`execution_allowed=false`、`credential_value_read_allowed=false`、`raw_endpoint_url_parsed=false`、`external_request_sent=false`、`write_commands_executed=false`、`load_restore_compact_executed=false` 和 `order_authoritative=false` 均稳定出现。`CHECKJSON` 只把 LOAD 分类为 write command，输出 `touches_wal=false`；本轮没有执行 LOAD，也没有修改 Store/WAL。最后一次 `BYE` 证明 CLI 正常退出。
