SMOKEJSON 大对象。截图内容来自同一份 `cmake-build-v1665` 干净构建和最终命令结果；原始临时

## 01-test-architecture.png

- 输入：`config/command-test-parts.txt`、`tests/command_test_suites.hpp`、
  `test_architecture_contract` 和十个 command suite 源文件。
- 结果：原 3805 非空行单体变为一个 15 非空行 runner 和九个职责 part，最大 part 576；
  457 个普通断言与 2447 个响应断言保持精确；体积 baseline 从 2 项收紧到 1 项。
- 意义：证明本版不是把长文件随意切开。共享 Store/processor/result 与原调用顺序保留，manifest
  中每个入口必须定义并调用一次，漏文件、漏调用、断言变化和新增超限文件都会失败。

  测试拆分没有通过删除历史断言或改写 fixture 获得局部绿灯。

## 04-cli-boundaries.png

- 输入：向本版新构建的真实 `minikv_cli.exe` 发送 `SMOKEJSON`、`CHECKJSON GET demo`、
  `HEALTH` 和 `QUIT`，对两份 prompt-prefixed JSON 做结构化解析。
- 结果：SMOKEJSON 的 `read_only=true`、`execution_allowed=false`、
  `order_authoritative=false`；CHECKJSON 的 `write_command=false`、`touches_wal=false`；HEALTH
  返回 OK，QUIT 返回 BYE。
- 意义：说明测试基础设施优化没有打开 router、跨项目写入、WAL、网络、凭据或执行权限。

## 截图方式和已记录偏差

截图通过 Playwright MCP 对本地临时 HTML 的四个独立证据区域生成。HTML 只承载已经由命令验证
过的摘要，不重新解释或模拟测试结果；截图结束后本地 HTTP server 与临时 HTML 一并清理。

拆分早期曾让每个 part 各建一份 fixture，`command_runtime_tests` 随即因 live key 状态不同失败；
最终设计改为共享 `CommandFixture&`，恢复旧单体的一次构造和原顺序，没有改期待值。全量验证早期
还发现四个手写 `add_test` 没有继承 MinGW runtime PATH，最终由统一 CMake helper 修复并在本次
353/353 中真实通过。CLI 归档脚本第一次又把 HEALTH 当成精确 `mini-kv> OK`；最终只匹配稳定
`OK live_keys=` 前缀，同时仍对两份 JSON 做严格字段解析。这些都是测试/验证基础设施修正，未修改
产品实现、fixture 或公共响应。
