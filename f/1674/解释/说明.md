# v1674 截图说明

## 归档原则

本版只保存四张能够说明架构变化、失败关闭能力、跨平台修复与九分生效条件的截图。
不把 45 万余字节的 `SMOKEJSON` 原文、完整 354 条 CTest 输出或构建二进制放入归档；
相应可复现命令和关键值集中在 `f/1674/证据/summary.txt`。

截图页面固定为 1400 x 848 CSS 像素，每个证据面板固定 1360 x 800。Playwright MCP
完成了页面导航、viewport 设置和四个面板的 overflow 检查，结果均为 `false`；但它的
截图调用分别在 5 秒和 30 秒超时，未生成文件。因此按仓库规则回退：前三张由
Chrome DevTools MCP 返回 PNG 后落入本目录；第四张时该 MCP 截图通道长时间无响应，
改用同一 HTML 页面和同一 viewport 的本机 Edge headless 确定性截图。临时 HTML、
base64 中转文件、浏览器 profile 与 HTTP 服务均在版本收尾时删除或停止。

## 01-product-source-architecture.png

这张图回答“为什么本版不是把根 CMake 的大段文字搬到另一个大文件”。
`config/product-sources.txt` 只拥有 316 个产品翻译单元的身份与顺序；
`cmake/MinikvSources.cmake` 拥有 schema、唯一性、存在性、main 排除和双向集合校验；
根 `CMakeLists.txt` 仍拥有 target 与构建策略。三者的职责边界分别是数据、校验、行为。

底部四个数字来自真实树：产品源 316，根 CMake 127 物理行，参与模块最大 257 行，
产品源/fixture diff 为 0。前、清单、后的三个文件均为 316 行、22154 字节和同一
SHA-256；因此“抽离”没有偷偷删源或重排链接输入。

## 02-fail-closed-parser.png

这张图说明 loader 不是只会处理 happy path。通过 CMake script 临时注入 CRLF、
重复项、错误扩展名、executable main、缺失文件和磁盘未登记源码，六类输入分别命中
明确错误。合法清单还必须与磁盘 `src/*.cpp` 减去五个 mains 的集合双向相等。

可复现入口：

```powershell
ctest --test-dir b1674 -C Debug -R test_architecture_contract --output-on-failure
```

合同自行建立临时红路径并在结束后清理。错误输入若意外成功，测试会失败；这使 316
不是文档数字，而是会阻止遗漏和偷渡的机械门。

## 03-cross-platform-test-order.png

这张图记录 v1673 远端红灯如何改变设计。macOS 证明解析生成器私有
`CTestTestfile.cmake` 文本并不跨平台；Windows 超时证明架构合同反复读取同一批文件
成本过高。v1674 让 `minikv_add_test()` 在真实注册点追踪名称，并把测试源统计改成
一次读取加缓存，不再依赖平台生成文件。

默认与 `MINIKV_BUNDLE_TESTS=OFF` legacy 配置都生成 354 个名称，逐行顺序完全相同；
LF 规范化名称序列 SHA-256 为
`f4ecf10465a185e4a3c04d5b0e0292ea47aba0a8d48e5992c7eb8d9159a3310e`。
最终本地全量为 354/354、37.71 秒。右上角只标注 `LOCAL GATE GREEN`，底部明确说明
跨平台分仍等待 v1674 精确提交的七条 CI，避免把 Windows 本机通过写成三平台通过。

## 04-nine-point-gate.png

这张图把“做到九分”变成条件式算式。合同、架构、数据建模、机械门、文档和范围克制
在本地合计 8.1；远端七条 CI 全绿后跨平台项才增加 1.0，使候选总分达到 9.1。
图中同时保留四条实证：354/354、真实 WAL 重开、CHECKJSON/SMOKEJSON no-write 与
no-execution、no-network 五类动作全为 false。

该图故意保留两句话：它不是外部 review，也不表示历史债务清零；CI 未绿时禁止宣布
到点。完整扣分和生效条件见 `f/1674/证据/nine-point-scorecard.md`。
