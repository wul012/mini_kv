# mini-kv v1651 命令归档说明

本版是 legacy credential-resolver 四回执的迁移前基线，不修改生产 formatter。归档只使用代码、测试、文档稳定后的最终命令结果；四张 PNG 均由 Playwright MCP 对最终结果摘要页面截图生成，没有使用早期构建或中途测试状态。

## 01-baseline-metrics

文件：`图片/v1651-01-baseline-metrics.png`

对应命令是新注册的 `runtime_receipt_legacy_resolver_chain_baseline_tests`。测试从四份 frozen release fixture 中提取完整 receipt 子对象，同时调用当前 public formatter，冻结对象字节数、顶层字段数和带 v1651 前缀的 FNV-1a 摘要。四份 fixture/runtime 必须逐字节一致，双方 `\\u0027` 次数都为零，因此没有新 canonical 或 whitespace waiver。

这张图还明确显示 census 仍为 14 builder-backed、13 pending、1 named waiver。v1651 新增的是 oracle，不是生产迁移，不能借测试数量增长虚报 builder 进度。

## 02-chain-and-census

文件：`图片/v1651-02-chain-and-census.png`

该图解释为何四个候选属于一个合理批次：它们按 v117-v120 连成 archive、blocked decision、plan intake、disabled candidate 四阶段，每个源文件恰好拥有一个 public formatter、独立 fixture 和领域测试。图中的 28 sources、27 formatter owners、14 builder-backed、13 pending、1 waiver 来自 `receipt_builder_census_contract` 的机械扫描，不是人工估算。

底部列出的字段是下一版迁移必须保持的共同边界。测试还分别要求四份对象自己的 `*_non_participation_receipt_only=true` 标记存在，防止一个对象被另一个同类模板错误替代。

## 03-final-tests

文件：`图片/v1651-03-final-tests.png`

聚焦命令在 `cmake-build-v1651` 中执行 census、通用 builder、v1651 baseline、四个领域 receipt、SMOKEJSON、runtime smoke 和 release manifest，共 10/10 通过。随后执行完整 `ctest --test-dir cmake-build-v1651 --output-on-failure`，347/347 通过，总耗时 105.18 秒。

完整套件包含 OSFS、Store、WAL、snapshot、RESP、command、历史 receipt 和文档契约。它证明新增测试与说明没有破坏项目其他区域；但它不把 mini-kv 描述成真实 credential resolver，也不代表跨项目实时联调。

## 04-cli-boundaries

文件：`图片/v1651-04-cli-boundaries.png`

真实输入顺序是 `SMOKEJSON`、`CHECKJSON LOAD data/prod.snap`、`QUIT`。SMOKEJSON 中四个目标 receipt marker 各出现一次；`read_only=true`、`execution_allowed=false`、`external_request_sent=false` 和 `write_commands_executed=false` 仍稳定出现。

`CHECKJSON LOAD` 只对命令做静态分类，输出 `command=LOAD` 与 `touches_wal=false`；本次没有执行 LOAD，没有替换 Store，没有追加 WAL。最后出现一次 `BYE`，进程正常退出，未留下 CLI/server 后台进程。

## 结论与清理

v1651 的输入是四份 frozen fixture、四个现有 public formatter 和只读命令列表；输出是一个不可与下一版生产改动互相迎合的 exact-byte 测试门。归档生成后关闭了 Playwright 页面；构建目录和临时日志在提交前删除，只保留本目录四张最终截图、此说明、`证据/summary.txt`、测试、讲解和维护文档。
