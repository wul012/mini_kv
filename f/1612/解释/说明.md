# mini-kv v1612 命令归档说明

## 本版范围

v1612 是 production-excellence K2 coverage baseline 的接线版本。本版新增 `MINIKV_COVERAGE`、Ubuntu coverage CI job、`gcovr` core coverage artifact，并用本地 CTest 保护 CI job 结构。它不新增运行时命令，不修改 WAL、snapshot、RESP、TCP、store 行为，不移动历史归档，不打开 router、write、WAL 或 execution 权限。

本地 Windows MinGW coverage 不是本版的 coverage source of truth。该工具链可以跑默认 Debug 构建和 CTest，但 gcov coverage sidecar 会因为仓库内长测试目标路径触发路径限制。本版将这个限制前置成 CMake 配置阶段的清晰失败，并把真实 coverage baseline 放到 GitHub Ubuntu CI。

## 截图清单

### 01-coverage-configure-fast-fail.png

命令：

```powershell
$env:PATH='D:\CLION\CLion 2025.1.1\bin\mingw\bin;' + $env:PATH
& 'D:\CLION\CLion 2025.1.1\bin\cmake\win\x64\bin\cmake.exe' -S . -B cmake-build-v1612-coverage -G 'MinGW Makefiles' -DCMAKE_BUILD_TYPE=Debug -DMINIKV_COVERAGE=ON
```

结果：配置阶段按预期失败，并输出 `MINIKV_COVERAGE writes gcov sidecar files whose paths exceed the bundled Windows MinGW limit in this repository. Use the Ubuntu coverage CI job.`。

意义：这证明 Windows MinGW 的 coverage 限制被明确收敛在配置阶段，不会让开发者误判为源码或测试失败。默认 Windows Debug 构建仍然受支持，coverage baseline 由 Ubuntu CI 生成。

### 02-default-configure-build.png

命令：

```powershell
& 'D:\CLION\CLion 2025.1.1\bin\cmake\win\x64\bin\cmake.exe' -S . -B cmake-build-v1612 -G 'MinGW Makefiles' -DCMAKE_BUILD_TYPE=Debug
& 'D:\CLION\CLion 2025.1.1\bin\cmake\win\x64\bin\cmake.exe' --build cmake-build-v1612 -- -j 4
```

结果：默认 Debug configure/build 成功。构建保留既有长 target 的 CMake path warning，但没有构建失败。

意义：这证明 `MINIKV_COVERAGE` 默认关闭且隔离，不会污染常规开发构建，也没有改变运行时命令、核心存储、WAL、snapshot、TCP/RESP 或 fixture 行为。

### 03-focused-ci-contract-test.png

命令：

```powershell
& 'D:\CLION\CLion 2025.1.1\bin\cmake\win\x64\bin\ctest.exe' --test-dir cmake-build-v1612 -C Debug -R dependabot_config_tests --output-on-failure --timeout 120 --progress
```

结果：`dependabot_config_tests` 1/1 通过。

意义：这条测试保护 CI workflow 的关键结构，包括 `ubuntu coverage` job、`gcovr` 安装、coverage configure/build/full CTest、core source filter、artifact 上传，以及禁止 `continue-on-error: true` 静默回归。

### 04-full-ctest.png

命令：

```powershell
& 'D:\CLION\CLion 2025.1.1\bin\cmake\win\x64\bin\ctest.exe' --test-dir cmake-build-v1612 -C Debug --output-on-failure --timeout 120 --progress
```

结果：334/334 CTest 通过，0 失败，总耗时约 166 秒。

意义：这证明 coverage baseline 接线没有破坏默认 Debug 路径下的 store、command、WAL、snapshot、RESP、TCP、fixture 和治理 evidence 测试。

## 下一步

推送 v1612 后，需要等待 GitHub Actions 的 `ubuntu coverage` job 产出 `mini-kv-core-coverage` artifact。下一版读取 `core-coverage.txt` 的真实 baseline，按 measured baseline minus 2 points 设置 `gcovr --fail-under-line`，再把 K2 标为 completed。只有这个 floor CI 绿后，才到 Claude review 的正确暂停点。
