# 第一批：项目定位与价值 Q&A

## Q01：老师问：你这个课程设计到底是什么？

答：这是一个教学版二级文件系统，入口程序是 `minikv_osfs`。它把一个二进制 `.img` 文件当成虚拟磁盘，在里面保存超级块、位图、用户表、inode 表、目录块和文件数据块。用户通过 `LOGIN` 进入系统，再用 `DIR`、`CREATE`、`OPEN`、`READ`、`WRITE`、`SEEK`、`TELL`、`FSCK` 等命令操作文件。重点不是做一个漂亮 shell，而是把文件系统的几层结构真正落到磁盘镜像里。

## Q02：老师问：它为什么适合操作系统课程设计？

答：因为它对应的是操作系统里文件系统这一块：磁盘空间管理、文件控制块、目录结构、用户权限、文件描述符、读写位移和一致性检查。课程要求一般不会只看命令能不能打印，而是要看有没有“磁盘如何组织、文件如何分配、目录如何查找、权限如何生效”的机制。这个项目把这些都做成可运行代码，并且有测试、演示脚本和 Linux 运行证据。

## Q03：老师问：输入和输出分别是什么？

答：输入有两类。第一类是命令，比如 `LOGIN alice alice123`、`CREATE report`、`WRITE 3 hello`、`FSCK`。第二类是磁盘镜像，也就是 `--disk osfs-course.img` 指定的二进制文件。输出也有两类：一类是命令返回，比如目录列表、文件内容、fd、offset、错误信息；另一类是镜像内部状态变化，比如 inode、bitmap、user table、data block 被写回，重开镜像以后还能读到。

## Q04：老师问：这个和原来的 mini-kv 项目有什么关系？

答：它在同一个仓库里，但不是拿 KV/WAL/TCP 主线来假装文件系统。OSFS 是独立入口、独立源码模块和独立测试，核心文件在 `src/osfs_*.cpp` 与 `include/minikv/osfs/` 下。原项目提供的是 C++ 工程基础和构建测试环境；课设功能本身围绕虚拟磁盘、inode、目录、权限、fd 和 FSCK 做出来。

## Q05：老师问：你怎么证明不是只写了一份报告？

答：有三层证据。第一层是源码，比如 `src/osfs_filesystem.cpp` 负责格式化和用户表初始化，`src/osfs_file_io.cpp` 负责 range I/O，`src/osfs_fsck.cpp` 负责一致性检查。第二层是测试，OSFS 聚焦测试和全量 CTest 都能跑。第三层是真实演示，`osfs-demo-script.txt` 可以驱动 `minikv_osfs` 生成 transcript，Linux 证据目录里也有构建、测试、demo、重开持久化的命令输出截图。

## Q06：老师问：项目最大的价值点在哪？

答：我认为价值在于它不是“命令模拟”，而是“结构可解释”。比如一个文件能从目录项追到 inode，再从 inode 的 direct 或 indirect 指针追到数据块；FSCK 还能反过来扫描这些结构，验证 bitmap 是否和可达块一致。老师现场问到任何一步，我都能从磁盘布局讲到代码路径，再讲到演示证据。

## Q07：老师问：你觉得难点是什么？

答：难点不是加命令，而是让命令背后的磁盘状态自洽。比如写文件时要同步更新数据块、inode size、mtime、block bitmap 和 superblock 空闲计数；删除或截断大文件时还要释放直接块、间接块和间接块指向的数据块；USERADD 既要写 user table，又要分配 UFD inode 和 MFD 目录项。只改一处很容易跑通单个命令，但 FSCK 和重开镜像会暴露问题。

## Q08：老师问：这个项目是不是工作量过大，和课程要求偏了？

答：范围是按课程“二级文件系统”收住的，没有去做任意深度目录、网络服务、生产密码系统、多进程并发或自动修复。加厚的部分集中在课程本身会考察的机制上：MFD/UFD、文件描述符位移、一级间接块、用户管理、权限和 FSCK。也就是说，工作量增加不是横向堆功能，而是把文件系统核心机制做实。

## Q09：老师问：如果只看行数，三千多行是不是也不算多？

答：文件系统课设不能只按行数判断。三千多行里有虚拟磁盘、磁盘布局、目录、文件 I/O、命令处理、FSCK、用户管理和测试，而且每一层都要和二进制镜像一致。比起堆上万行界面代码，老师更可能关心一次 `WRITE` 改了哪些块、一次 `FSCK` 检查哪些关系、重开镜像是否还在。这些机制已经覆盖了课程重点。

## Q10：老师问：为什么还要做 Linux 运行证据？

答：题目语境是 Linux 二级文件系统。虽然代码是 C++ 跨平台的，Windows 本地也能跑，但 Linux 证据能补上运行环境口径。`linux-run-evidence-20260706/` 里记录了 Linux 主机上的工具链、CMake 构建、OSFS 测试、demo 脚本、重开持久化和清理检查，说明它不是只在开发机偶然跑通。

## Q11：老师问：如果现场只能讲一分钟，你怎么介绍？

答：我会先说“这是一个基于二进制镜像的二级文件系统”，然后画出 superblock、bitmap、user table、inode table、data region。接着用 `CREATE + WRITE` 举例：登录拿到 UFD，创建目录项和 inode，写文件分配数据块，超过 8 个直接块后使用一级间接块。最后说 FSCK 会反向扫描这些结构，检查 bitmap、目录和用户表是否一致。

## Q12：老师问：你最应该主动承认的边界是什么？

答：要主动说清四个边界：第一，它是 MFD/UFD 两级目录，不是任意深度目录树；第二，它做了一级间接块，没有做二级和三级间接块；第三，FSCK 是只读检查，不自动修复；第四，密码哈希是教学用途，不等同生产安全方案。主动说边界反而能说明自己知道项目做到哪、没做到哪。
