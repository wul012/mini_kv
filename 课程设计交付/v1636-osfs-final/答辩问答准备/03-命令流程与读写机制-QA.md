# 第三批：命令流程与读写机制 Q&A

## Q25：老师问：未登录时为什么不能操作文件？

答：命令处理器维护当前登录用户状态。`WHOAMI` 可以在未登录时查看状态，但 `CREATE`、`OPEN`、`DIR` 等文件命令需要当前用户 uid。未登录直接操作会返回 `ERR login required`。这避免了绕过 user table 和权限检查。

## Q26：老师问：LOGIN 的过程是什么？

答：`LOGIN user password` 会读取 user table，按用户名找到记录，再用同样的教学哈希函数计算密码哈希并比较。成功后命令处理器保存当前用户的 username、uid、gid 和 UFD inode。登录切换时会清空已有 fd，避免一个用户拿着另一个用户打开过的文件句柄继续读写。

## Q27：老师问：CREATE 会修改哪些结构？

答：`CREATE name` 会先定位当前用户 UFD，检查文件名是否合法、是否重名；然后从 inode bitmap 分配一个空 inode，初始化文件 inode 的 type、mode、owner、时间戳；再在 UFD 目录块里写入 name 到 inode 的目录项；最后写回 UFD、inode table、inode bitmap、superblock。

## Q28：老师问：OPEN 为什么返回 fd，而不是直接读写文件名？

答：fd 用来模拟操作系统里的打开文件表。打开文件后，句柄里保存文件名、模式、read offset 和 write offset。这样 `READ fd`、`WRITE fd`、`SEEK fd offset`、`TELL fd` 才有意义。否则每次读写只按文件名操作，就体现不出文件描述符和位移这部分课程内容。

## Q29：老师问：READ fd [length] 怎么决定读哪里？

答：它看 fd 里的 read offset，从这个偏移开始读。如果提供 length，就读指定长度；不提供就读到文件末尾。读完以后 read offset 会向后推进。底层是 `read_file_range`，会根据 offset 算出逻辑块号和块内偏移，再通过 inode 的 direct 或 indirect 指针找到真实数据块。

## Q30：老师问：WRITE fd text 写到哪里？

答：它看 fd 的 write offset，从该位置开始覆盖或扩展文件。写入后 write offset 会增加。底层 `write_file_range` 会先做权限检查，再计算目标文件大小需要多少块，必要时分配数据块；写入跨块时会分段写入多个 block。最后更新 inode size、mtime、bitmap 和 superblock 空闲计数。

## Q31：老师问：SEEK 和 TELL 有什么作用？

答：`SEEK fd offset` 同时调整该 fd 的读写位移，`TELL fd` 显示当前 read_offset 和 write_offset。它们让文件描述符状态可见，也方便演示随机读写。例如演示里把 bigfile seek 到 4088，再读 64 字节，正好跨过直接块边界进入间接块数据。

## Q32：老师问：OPEN 的 r、w、rw、a 模式有什么区别？

答：`r` 只读，不能写；`w` 用于写入并会截断原文件；`rw` 保留原内容，可以定位更新；`a` 是追加，write offset 从文件末尾开始。这样能体现打开模式对后续 fd 行为的影响，也能测试权限和 truncate 是否生效。

## Q33：老师问：一级间接块是怎么工作的？

答：一个 inode 先有 8 个直接块指针。默认 block size 是 512，所以 8 个直接块最多覆盖 4096 字节。超过这个容量时，inode 的 `indirect_block` 会指向一个“索引块”，这个索引块里按 `uint32_t` 数组保存后续数据块号。读写第 9 个及之后的数据块时，先读 indirect block，再根据逻辑块号取出真实数据块号。

## Q34：老师问：怎么证明间接块真的参与 I/O？

答：演示写了 4291 字节的大文件，超过 4096 字节直接块边界。然后从 4088 偏移开始读 64 字节，返回内容跨过 `DIRECT8|` 后进入 `INDIRECT-BEGIN|`。如果 indirect block 只是字段而没有参与读写，这个跨边界读不可能字节级正确。测试还覆盖了超出上限拒绝和删除大文件释放间接块。

## Q35：老师问：删除文件时会不会只删目录项，不释放块？

答：不会。删除会定位目录项和 inode，检查权限，然后释放 inode 关联的存储。这里不仅释放 direct block，也会处理 `indirect_block`：先读取间接块里的数据块号释放后续数据块，再释放间接块本身，最后释放 inode bitmap 并更新 superblock 空闲计数。

## Q36：老师问：为什么写文件前要先检查空间？

答：因为文件系统必须避免写到一半才发现空间不够，导致 inode、bitmap 和数据块状态不一致。实现里会先算目标大小需要的数据块数，以及是否需要额外的 indirect block；同时把旧文件可回收块计入判断。空间够才释放旧存储并重新分配，空间不够直接返回错误。
