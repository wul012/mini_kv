# 第二批：磁盘布局与目录结构 Q&A

## Q13：老师问：你的虚拟磁盘怎么组织？

答：虚拟磁盘是一个固定块大小的二进制文件。前面是元数据区：Block 0 是 superblock，后面有 block bitmap、inode bitmap、user table，再后面是 inode table。data region 里放 MFD/UFD 目录块、普通文件数据块和间接块。程序不是把这些存在内存对象里就结束，而是通过 `VirtualDisk::read_block/write_block` 按块读写 `.img`。

## Q14：老师问：superblock 里保存什么？

答：superblock 保存文件系统几何和全局状态，比如版本号、block size、block count、inode count、bitmap 位置、user table 位置、inode table 位置、data region 起始块、MFD inode、free block count、free inode count、user count。打开镜像时先读 superblock，才能知道后续区域怎么解释。

## Q15：老师问：为什么要有 block bitmap？

答：block bitmap 负责表示哪些块已经被占用。格式化时，元数据块和初始目录块会被标记占用；写文件、创建 UFD 或分配 indirect block 时会找空闲位并置 1；删除、truncate 或覆盖释放旧数据时会把对应位清掉。FSCK 会重新计算一份“应该占用的 block bitmap”，再和磁盘上的 bitmap 比较。

## Q16：老师问：inode bitmap 的作用是什么？

答：inode bitmap 管 inode 表里哪些槽位正在使用。创建文件或新用户 UFD 时要分配 inode，删除文件时要释放 inode。它和 block bitmap 分开，是因为 inode 是文件元数据编号，block 是磁盘空间编号，两者生命周期不完全一样。

## Q17：老师问：inode 里有哪些关键字段？

答：inode 里有 used、type、mode、owner_uid、owner_gid、size、ctime/atime/mtime、8 个 direct block 指针和 1 个 indirect block 指针。目录 inode 的 direct 指针指向目录项块；普通文件 inode 的 direct/indirect 指针指向文件数据块。权限和大小都在 inode 里，不是临时算出来的。

## Q18：老师问：MFD 和 UFD 是怎么实现的？

答：MFD 是主文件目录，里面记录用户名到 UFD inode 的映射；每个用户还有自己的 UFD，UFD 里面记录该用户文件名到文件 inode 的映射。格式化时会给 root、alice、bob 分配各自 UFD；`USERADD carol` 时也会分配新的 UFD inode 和目录数据块，并把 Carol 写进 user table 和 MFD。

## Q19：老师问：怎么证明不是一个目录用 owner 过滤？

答：创建用户时会给每个用户分配独立 UFD inode 和独立目录块，user table 中保存 `ufd_inode`，MFD 中也有同名目录项指向该 UFD。`DIR` 默认读当前用户 UFD，root 执行 `DIR alice` 时通过 MFD/user table 找到 Alice 的 UFD。Bob 的 DIR 为空不等于过滤出来为空，而是 Bob 的 UFD 本身没有那些目录项。

## Q20：老师问：user table 为什么需要单独存在？

答：MFD 只能表达“用户名对应哪个 UFD inode”，但登录还需要密码校验、uid/gid 和用户状态。user table 把 username、uid、gid、ufd_inode、password_hash 持久化起来。这样 `LOGIN`、`USERADD`、`PASSWD` 和 FSCK 都能基于同一份磁盘用户数据工作。

## Q21：老师问：目录项里存什么？

答：目录项主要存 used、name 和 inode 编号。它本身不存完整文件内容，只负责从名字找到 inode。文件大小、权限、owner、时间戳和数据块指针在 inode 中。这个设计接近真实文件系统的思路：目录是名字到 inode 的索引。

## Q22：老师问：为什么要有 free block count 和 free inode count？

答：它们是 superblock 里的全局统计，方便快速了解剩余空间，也能让 FSCK 检查一致性。每次分配或释放 block/inode 后都要重新计算并写回。FSCK 会根据重新扫描得到的期望 bitmap 计算 free count，如果和 superblock 不一致，就说明元数据已经漂移。

## Q23：老师问：EXT2 里有 group descriptor，你这里为什么没有？

答：这里主动标成教学简化。课程要求的重点是二级目录、磁盘空间管理、FCB/inode、权限和读写，不是完整复刻 EXT2。当前实现相当于单教学 block group：superblock 直接记录各区域位置，不再拆多个 block group。这样范围更适合课设，同时不掩盖真实边界，需求矩阵里也写成“部分/教学简化”。

## Q24：老师问：重开镜像后为什么还能看到文件？

答：因为目录、inode、bitmap、user table 和数据块都写到了 `.img`。程序重开时不是恢复内存快照，而是先读 superblock，再根据 superblock 读各个区域。比如 Carol 的新密码、Carol 的 UFD、Alice 的 bigfile 都可以在去掉 `--format` 后重新打开镜像继续访问。
