## 关键解释

### 1. DIR 字段已经直接展示

```text
osfs> DIR
name inode physical mode owner size
delete-me 6 22 0644 1002 9
```

这一段把 `DIR` 的表头打出来了，字段含义比较直观：

- `name`：文件名。
- `inode`：文件对应的 inode 编号。
- `physical`：文件数据所在的物理块位置。
- `mode`：保护码，也就是权限位。
- `owner`：文件所有者 uid。
- `size`：文件长度。

这里 `delete-me` 的 `size` 是 `9`，正好对应写入内容 `DELETE-ME` 的 9 个字符；`mode` 是 `0644`，说明所有者可读写，其他用户只读。这个输出可以直接对应课程要求里“目录显示文件名、物理地址、保护码和文件长度”的部分。

### 2. DELETE 删除文件

```text
osfs> DELETE delete-me
OK deleted delete-me
osfs> DIR
(empty)
```

`DELETE delete-me` 返回成功后，再执行 `DIR` 显示空目录，说明 `delete-me` 的目录项已经被移除。

### 3. 删除后不能再打开

```text
osfs> OPEN delete-me r
ERR file not found
```

删除后再按原文件名打开，系统返回 `ERR file not found`。这比只看 `DIR` 为空更稳一些，因为它说明文件名查找也已经找不到该文件。

### 4. 写权限拒绝

```text
osfs> CREATE secret
OK created secret
osfs> CHMOD secret 0600
OK chmod secret 0600
osfs> DIR
name inode physical mode owner size
secret 6 0 0600 1002 0
osfs> LOGIN other other123
OK login other uid=1003
osfs> OPEN deluser/secret w
ERR permission denied
osfs> OPEN deluser/secret rw
ERR permission denied
```

`secret` 的权限被设置为 `0600`，只有所有者 `deluser` 可以读写。另一个用户 `other` 分别用 `w` 和 `rw` 打开，都被拒绝。这补上了前面 Step 5 主要展示“读权限拒绝”之外的“写权限拒绝”证据。

### 5. 删除和权限测试后 FSCK 仍通过

```text
osfs> FSCK
FSCK OK checks=6
```

删除文件、再次查找、创建私有文件并测试写权限之后，`FSCK` 仍然通过。它说明目录、inode、位图、用户表等结构在这一轮操作后仍保持一致。
