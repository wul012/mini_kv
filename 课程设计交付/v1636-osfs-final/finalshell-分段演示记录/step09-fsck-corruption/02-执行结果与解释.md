`STAT` 给出了本次运行中的真实位置：文件使用 inode 4，首个数据块是物理块 20，文件长度为 14 字节。破坏前先运行 FSCK 并得到 `OK`，这样后续错误才能归因于位图修改，而不是原镜像已有问题。

脚本从 `STAT` 日志解析物理块号，再清除 block bitmap 中对应的位：

```text
azureuser@myvps:~/mini-kv$
azureuser@myvps:~/mini-kv$ DATA_BLOCK=$(sed -n 's/.*physical=\([0-9][0-9]*\).*/\1/p' /tmp/osfs-step09-before.log | head -n 1)
azureuser@myvps:~/mini-kv$ test -n "$DATA_BLOCK"
azureuser@myvps:~/mini-kv$ echo "corrupting_data_block=$DATA_BLOCK"
corrupting_data_block=20
azureuser@myvps:~/mini-kv$
azureuser@myvps:~/mini-kv$ python3 - /tmp/osfs-step09.img "$DATA_BLOCK" <<'PY'
> from pathlib import Path
> import sys
>
> path = Path(sys.argv[1])
> block = int(sys.argv[2])
> image = bytearray(path.read_bytes())
> block_size = 512
> block_bitmap_block = 1
> offset = block_bitmap_block * block_size + block // 8
> image[offset] &= ~(1 << (block % 8))
> path.write_bytes(image)
> print(f"cleared block bitmap bit for block {block} at byte offset {offset}")
> PY
cleared block bitmap bit for block 20 at byte offset 514
```

块号没有写死。脚本先取得 `DATA_BLOCK=20`，再按 `512 + 20 // 8` 算出位图字节偏移 514，并清除该字节内代表物理块 20 的位。文件 inode 和数据内容没有被改写，此时磁盘位图却把仍被引用的数据块标成了空闲。
