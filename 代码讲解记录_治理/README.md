# 代码讲解记录治理规则

本目录记录 mini-kv 代码讲解文档的治理规则。目标不是重写历史 900 多篇讲解，而是先把入口、模板和未来增量管住，避免继续把文档堆进一个巨大目录或巨大 README。

## 本轮判断

当前问题：

- `代码讲解记录_生产雏形阶段_第三册/` 已有 774 篇 Markdown。
- 第三册 `README.md` 接近 196KB，已经不适合作为唯一入口。
- 历史文件名很长，部分主题相近，靠一个 README 顺序查找成本太高。
- 不适合一次性搬迁全部历史文件；这种操作会制造大量 rename diff，后续追溯更困难。

本轮处理：

- 新增总索引：`代码讲解记录_总索引.md`。
- 新增第三册分段入口：`代码讲解记录_生产雏形阶段_第三册/README_分段入口.md`。
- 启用第四册：`代码讲解记录_生产雏形阶段_第四册/`。
- 新增统一模板：`production-readiness-walkthrough-template.md`。
- 记录批次计划：`批次整治记录.md`。

## 参考模式

后续 mini-kv 讲解优先参考：

```text
D:\nodeproj\orderops-node\代码讲解记录\107-production-readiness-summary-v3-v103.md
```

只参考表达结构，不复制 Node 项目内容。它的关键模式是：

- 先说目标、角色和“不是什么”。
- 再说入口在哪里。
- 再说响应模型或证据结构。
- 再说上游证据、配置或依赖关系。
- 再说核心流程如何聚合。
- 再说边界字段控制面应该怎么理解。
- 最后讲测试覆盖保护什么行为，而不只是罗列命令。

## 可省略原则

可以没有代码讲解。以下场景默认不硬写：

- 只改索引、拼写、链接或归档说明。
- 没有新增入口、响应结构、运行语义、测试策略或边界字段。
- 讲解只能重复 commit message，不能给后续维护者增加理解价值。

但只要写了讲解，就必须按模板写，不能再出现只堆文件名、字段名、命令输出的流水账。

## 分册原则

- 第三册已归档，不再追加新讲解。
- 第四册作为当前活跃册，从 v1565 之后的新讲解开始使用。
- 每册 README 只保留短索引，不再写成巨型目录。
- 历史文件不主动移动；需要降低查找成本时，新增分段索引或主题索引。
- 如果第四册以后超过 150 篇或 README 超过 50KB，优先开第五册。

## 质量门槛

新讲解必须回答这些问题：

- 本版为什么存在？
- 它不是什么，哪些权限没有打开？
- 入口命令、模块或文件在哪里？
- 数据结构、响应字段或 evidence 字段如何理解？
- 核心聚合或格式化流程在哪里？
- `read_only`、`execution_allowed`、`order_authoritative`、`mutates_store`、`touches_wal`、`warnings`、`blockers`、`diagnostics` 等字段如何被控制面消费？
- 每类测试保护什么行为？
- CLI/TCP/fixture/screenshot/CI 证据各自证明什么？
- 一句话说明它给 Node/Java/mini-kv 松耦合融合带来的价值。
