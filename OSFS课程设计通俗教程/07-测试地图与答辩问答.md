# 测试地图、演示证据与答辩问答

“有测试”这句话信息量很低。老师真正关心的是：哪一项要求由哪条自动断言保护，是否在真实程序中跑过，失败和损坏路径有没有覆盖。OSFS 的证据应分层看。

## 四类证据不要混用

| 类型 | 当前位置 | 能证明什么 | 不能单独证明什么 |
|---|---|---|---|
| C++ focused test | `tests/osfs_tests.cpp`、`tests/osfs_resilience_tests.cpp` | API、命令、磁盘重开、异常分支会自动断言 | 不能展示远端 Linux 操作过程 |
| CLI smoke | `osfs_cli_smoke` | 真实 `minikv_osfs` 可执行程序和脚本入口可用 | 场景较短，不覆盖所有机制 |
| Linux 实跑 | `linux-run-evidence-20260706/`、FinalShell 分段记录 | Linux 工具链、构建、运行和真实输出 | 截图本身不会在回归时自动失败 |
| 报告与需求矩阵 | `课程设计交付/v1636-osfs-final/` | 帮助老师按课程要求查证 | 文档声明不能代替源码和测试 |

最强的组合通常是：先指出源码机制，再给自动测试名称，最后展示一段真实命令输出。FSCK 还可以追加故障注入，证明 ERROR 分支。

## 原有核心测试地图

`osfs_tests` 一个 CTest 目标中包含七个场景：

| 场景函数 | 保护的机制 |
|---|---|
| `test_disk_users_and_two_level_directories` | 磁盘格式 v3、预置用户、不同 UFD inode、登录、目录隔离、0600、root 审查、重开持久化、删除 |
| `test_authenticated_command_shell` | 未登录拒绝、错误密码、命令入口、fd 基本读写、跨用户权限、root 读取 |
| `test_descriptor_offsets_and_range_io` | 连续偏移、SEEK/TELL、rw 覆盖、append、EOF、跨 512 字节块、直接到间接边界、上限拒绝、截断、重开 |
| `test_failed_rewrite_preserves_existing_data` | 空间不足时覆盖失败但旧内容不丢 |
| `test_indirect_blocks_persist_and_release` | direct/indirect 边界、重开、删除后数据块和间接块全部释放 |
| `test_fsck_reports_ok_and_detects_corruption` | 干净 FSCK、block bitmap 人为损坏、命令级 ERROR 输出 |
| `test_useradd_passwd_and_fsck_command` | root-only USERADD、重复用户、PASSWD、自改与越权拒绝、重开认证、用户表满、FSCK |

这些场景已经超过“命令能跑”的最小课设，但审计后仍发现几条高价值分支没有直接断言，所以本轮没有停在文档改写。

## 新增 resilience 测试地图

`osfs_resilience_tests` 作为独立 CTest 目标，避免继续膨胀原测试文件：

| 场景函数 | 为什么值得补 |
|---|---|
| `test_login_switch_closes_descriptors` | 防止 Alice 的旧 fd 在 Bob 会话继续使用；同时验证 r/a 模式和重复 CLOSE |
| `test_group_permissions_and_root_override` | 原测试主要覆盖 owner/other，本场景真正走 group 位，并区分“可写”与“可删除”；root 仍可管理 |
| `test_directory_growth_and_slot_reuse` | 512/64=8，精确跨过第九个目录项，证明 UFD 会扩块；删除后新文件复用槽位，不继续浪费块 |
| `test_sparse_range_write_zero_fills_gap` | 从空文件 offset 600 写入，检查前 600 字节为零、跨块读取正确、重开仍在 |
| `test_fsck_finds_inode_bitmap_corruption` | 在已有 block bitmap 故障外，再证明 inode bitmap 独立漂移会进入 FSCK ERROR |

这五条不是为了增加行数。它们分别关闭会话安全、权限分支、目录容量、区间 I/O 边界和一致性诊断的证据缺口。

## 怎么复现

Windows 本地使用 CLion 工具链：

```powershell
$env:PATH='D:\CLION\CLion 2025.1.1\bin\mingw\bin;' + $env:PATH
ctest --test-dir cmake-build-v1663 `
  -R '^(osfs_tests|osfs_resilience_tests|osfs_cli_smoke)$' `
  --output-on-failure
```

Linux/FinalShell 使用构建目录中的程序：

```bash
ctest --test-dir build-osfs-linux -R '^(osfs_tests|osfs_resilience_tests|osfs_cli_smoke)$' --output-on-failure
```

正式交付前仍要跑全量 CTest，因为 OSFS 复用同一个 `minikv` 静态库和 CMake 配置。focused test 负责快速定位，full test 负责证明没有破坏 KV/WAL/TCP 等既有目标。

## 老师最可能追问的十个问题

**1. 这是不是把文件内容存在 `map<string,string>` 里？**

不是。文件内容按物理块写入 `.img`；目录项、inode、位图和用户表都有固定落盘结构，重开镜像后可以恢复。

**2. 为什么说是真二级目录，不是按 owner 过滤？**

每个用户记录和 MFD 项都指向独立 UFD inode，不同 UFD 又有各自目录数据块。测试直接要求 Alice/Bob 的 `ufd_inode` 不同。

**3. fd 和 inode 有什么区别？**

inode 是磁盘上的文件身份和元数据；fd 是当前命令处理器内存里的打开句柄，带模式和偏移。登录切换会清 fd，但 inode 和内容仍在。

**4. 第九个数据块怎样找到？**

inode 的八个 direct 用完后，`indirect_block` 指向一个 128 项块号数组，数组第 0 项对应逻辑块 8。

**5. FSCK 为什么可信？**

它从 used inode、目录和用户关系重建期望位图，再和磁盘位图比较。测试分别破坏 block bitmap 和 inode bitmap，都会得到针对性 ERROR。

**6. WRITE 失败会不会毁掉旧文件？**

整文件覆盖会先预检“空闲块 + 当前文件可回收块”是否够用。空间不足测试要求失败后旧内容仍逐字节不变。

**7. 0640 对 Bob 有什么效果？**

Alice 和 Bob 的 gid 都是 100。Bob 能通过 group read 读取，但没有 group write 时不能写；改成 0660 后可以写，但删除仍要求 owner 或 root。

**8. 空文件为什么 physical 是 0？**

CREATE 只分配 inode 和目录项。第一次 WRITE 才分配数据块，所以空文件没有首数据块。

**9. 密码安全吗？**

它是课程演示用的固定 64 位摘要，不保存明文，但不是生产密码哈希。项目明确披露这一限制。

**10. 为什么不做任意深度目录和自动修复？**

题目重点是 MFD/UFD 二级目录和核心分配机制。自动修复、任意目录树、崩溃日志和多进程一致性会引入另一层设计问题，本课设如实列为边界。

更完整的几十组问题和速记图见 [答辩问答准备](../课程设计交付/v1636-osfs-final/答辩问答准备/README.md)。答辩前不要只背答案，至少亲手走一遍 `CREATE -> inode -> block` 和 `FSCK -> 重建期望位图 -> mismatch` 两条链。
