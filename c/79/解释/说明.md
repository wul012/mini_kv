# v79 运行调试归档说明

本轮先只读检查了 `D:\nodeproj\orderops-node\docs\plans\README.md` 和最新具体计划 `D:\nodeproj\orderops-node\docs\plans\v197-post-readiness-checkpoint-roadmap.md`。计划要求 mini-kv v79 增强 `SMOKEJSON` 的 operator-window consumer / identity-neutral proof 字段，让 Node v198/v200 确认 mini-kv 只提供 runtime evidence，不接收或伪造业务身份；该范围合理，本轮按此执行。

## 截图说明

1. `01-cmake-configure.png`
   - 命令：`cmake -S . -B cmake-build-v79 -G "MinGW Makefiles" -DCMAKE_BUILD_TYPE=Debug`
   - 结果：配置成功，生成 `cmake-build-v79`。
   - 意义：确认 v79 版本号、CMake target 和本机 CLion/MinGW 工具链能正常进入构建。

2. `02-cmake-build.png`
   - 命令：`cmake --build cmake-build-v79 --parallel 2`
   - 结果：`minikv` 静态库、CLI、server、client、benchmark 和全部测试目标构建成功。
   - 意义：确认新增 `operator_window` 字段没有破坏 C++ 编译或链接。

3. `03-ctest.png`
   - 命令：`ctest --test-dir cmake-build-v79 --output-on-failure`
   - 结果：31/31 tests passed。
   - 意义：确认静态 fixture、release manifest、runtime `SMOKEJSON` 输出和历史版本断言全部对齐到 v79。

4. `04-real-smoke.png`
   - 命令：启动 `minikv_server 6417 127.0.0.1`，用 `minikv_client 127.0.0.1 6417 5000 --connect-retries 10 --retry-delay-ms 100` 执行 `SMOKEJSON`、`INFOJSON`、`STORAGEJSON`、`HEALTH`、`GET restore:real-read-token`、`QUIT`。
   - 结果：`SMOKEJSON` 返回 `version:"0.79.0"`、`operator_window.identity_neutral_proof:true`、`taxonomy_digest:"fnv1a64:f92fcba55feb26a2"`；`GET restore:real-read-token` 返回 `(nil)`。
   - 意义：证明 v79 的身份中性证明来自真实 TCP 只读窗口，不需要执行 `LOAD`、`COMPACT`、`SETNXEX` 或 restore。

## 边界

- 本轮没有修改 Node、Java 或 aiproj。
- 本轮没有读取生产密钥、没有连接生产数据库、没有打开 `UPSTREAM_ACTIONS_ENABLED`。
- mini-kv 仍然不是 Java 订单权威存储。
- `operator_window.identity_neutral_proof=true` 只是身份中性 runtime evidence，不是认证、授权或生产操作者身份凭据。
- 验证结束后 v79 server 已停止，没有留下后台进程。
